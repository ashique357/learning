# Kubernetes Policies 📋

## What are Kubernetes Policies? (Simple Explanation)

Think of Kubernetes policies like **city regulations and rules**!

### Real-Life Example 🏙️
In a city, you have different types of rules:
- **Budget Limits (Resource Quotas)** - How much each department can spend
- **Building Codes (Limit Ranges)** - Size limits for new buildings
- **Emergency Procedures (Pod Disruption Budgets)** - How many services can be down during maintenance
- **Zoning Laws (Network Policies)** - What can be built where
- **Safety Regulations (Pod Security Policies)** - Safety standards for all buildings
- **Quality Standards (Admission Controllers)** - Requirements before approval

**Kubernetes policies work the same way!**

## 1. Resource Quotas 💰 (Department Budgets)

### What are Resource Quotas?
**Spending limits for namespaces** - like department budgets in a company

**Real-Life Example:** Company department budgets
- **Development Team** - $10,000/month budget for servers
- **Production Team** - $50,000/month budget (more critical)
- **Testing Team** - $5,000/month budget (temporary use)
- **Can't exceed budget** - Prevents overspending

### Basic Resource Quota Example
```yaml
# department-budgets.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-team-budget
  namespace: development
spec:
  hard:
    # Compute resources
    requests.cpu: "10"          # Total CPU requests: 10 cores
    requests.memory: 20Gi       # Total memory requests: 20GB
    limits.cpu: "20"            # Total CPU limits: 20 cores
    limits.memory: 40Gi         # Total memory limits: 40GB

    # Storage resources
    requests.storage: 100Gi     # Total storage requests: 100GB
    persistentvolumeclaims: "10" # Max 10 PVCs

    # Object counts
    pods: "50"                  # Max 50 pods
    services: "10"              # Max 10 services
    secrets: "20"               # Max 20 secrets
    configmaps: "30"            # Max 30 configmaps

    # Specific resource types
    count/deployments.apps: "20"     # Max 20 deployments
    count/jobs.batch: "5"            # Max 5 jobs
    count/cronjobs.batch: "3"        # Max 3 cronjobs
```

### Production vs Development Quotas
```yaml
# production-budget.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: production-budget
  namespace: production
spec:
  hard:
    requests.cpu: "50"          # Production gets more resources
    requests.memory: 100Gi
    limits.cpu: "100"
    limits.memory: 200Gi
    pods: "200"
    services: "50"
---
# testing-budget.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: testing-budget
  namespace: testing
spec:
  hard:
    requests.cpu: "5"           # Testing gets fewer resources
    requests.memory: 10Gi
    limits.cpu: "10"
    limits.memory: 20Gi
    pods: "20"
    services: "5"
```

### Step-by-Step: Set Department Budgets
```bash
# Step 1: Create namespaces for different teams
kubectl create namespace development
kubectl create namespace production
kubectl create namespace testing

# Step 2: Apply resource quotas
kubectl apply -f department-budgets.yaml
kubectl apply -f production-budget.yaml
kubectl apply -f testing-budget.yaml

# Step 3: Check quotas
kubectl get resourcequota -A

# Expected output:
NAMESPACE     NAME                AGE
development   dev-team-budget     1m
production    production-budget   1m
testing       testing-budget      1m

# Step 4: See quota details
kubectl describe resourcequota dev-team-budget -n development

# Step 5: Test quota limits - try to exceed budget
kubectl create deployment big-app --image=nginx --replicas=60 -n development
# Should fail after 50 pods due to quota!

# Step 6: Check quota usage
kubectl describe resourcequota dev-team-budget -n development
# Shows: Used: 50, Hard: 50 for pods
```

## 2. Limit Ranges 🏗️ (Building Codes)

### What are Limit Ranges?
**Size limits for individual resources** - like building codes that set min/max sizes

**Real-Life Example:** City building codes
- **Minimum house size** - 500 sq ft (prevents tiny unsafe houses)
- **Maximum house size** - 5000 sq ft (prevents mansions in small neighborhoods)
- **Default house size** - 1200 sq ft (if not specified)
- **Applies to each building** - Not total neighborhood

### Basic Limit Range Example
```yaml
# building-codes.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: building-codes
  namespace: development
spec:
  limits:
    # Rules for Pods
    - type: Pod
      max:
        cpu: "2"                  # No pod can use more than 2 CPU
        memory: "4Gi"             # No pod can use more than 4GB RAM
      min:
        cpu: "100m"               # Every pod must request at least 0.1 CPU
        memory: "128Mi"           # Every pod must request at least 128MB RAM

    # Rules for Containers
    - type: Container
      default:                    # Default if not specified
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:             # Default requests if not specified
        cpu: "250m"
        memory: "256Mi"
      max:
        cpu: "1"                  # No container can use more than 1 CPU
        memory: "2Gi"             # No container can use more than 2GB RAM
      min:
        cpu: "50m"                # Every container must request at least 0.05 CPU
        memory: "64Mi"            # Every container must request at least 64MB RAM

    # Rules for Persistent Volume Claims
    - type: PersistentVolumeClaim
      max:
        storage: "50Gi"           # No PVC can request more than 50GB
      min:
        storage: "1Gi"            # Every PVC must request at least 1GB
```

### Different Limit Ranges for Different Environments
```yaml
# production-building-codes.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: production-codes
  namespace: production
spec:
  limits:
    - type: Container
      default:
        cpu: "1"                  # Production gets bigger defaults
        memory: "1Gi"
      defaultRequest:
        cpu: "500m"
        memory: "512Mi"
      max:
        cpu: "4"                  # Higher limits for production
        memory: "8Gi"
      min:
        cpu: "100m"               # Higher minimums for production
        memory: "256Mi"
---
# testing-building-codes.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: testing-codes
  namespace: testing
spec:
  limits:
    - type: Container
      default:
        cpu: "200m"               # Testing gets smaller defaults
        memory: "256Mi"
      defaultRequest:
        cpu: "100m"
        memory: "128Mi"
      max:
        cpu: "500m"               # Lower limits for testing
        memory: "1Gi"
      min:
        cpu: "50m"                # Lower minimums for testing
        memory: "64Mi"
```

### Step-by-Step: Implement Building Codes
```bash
# Step 1: Apply limit ranges to namespaces
kubectl apply -f building-codes.yaml
kubectl apply -f production-building-codes.yaml
kubectl apply -f testing-building-codes.yaml

# Step 2: Check limit ranges
kubectl get limitrange -A

# Step 3: See limit range details
kubectl describe limitrange building-codes -n development

# Step 4: Test with pod that doesn't specify resources
kubectl run test-pod --image=nginx -n development

# Step 5: Check if defaults were applied
kubectl describe pod test-pod -n development
# Should show default CPU: 500m, Memory: 512Mi

# Step 6: Try to create pod that exceeds limits
kubectl run big-pod --image=nginx -n development --requests=cpu=3,memory=6Gi
# Should fail: exceeds maximum resource limits

# Step 7: Try to create pod below minimum
kubectl run tiny-pod --image=nginx -n development --requests=cpu=10m,memory=32Mi
# Should fail: below minimum resource requirements
```

## 3. Pod Disruption Budgets 🚨 (Emergency Procedures)

### What are Pod Disruption Budgets?
**Availability rules during maintenance** - like emergency procedures that ensure critical services stay running

**Real-Life Example:** Hospital emergency procedures
- **ICU** - Must keep at least 80% of beds available during maintenance
- **Emergency Room** - Must keep at least 2 doctors on duty always
- **Pharmacy** - Can close temporarily but not during peak hours
- **Prevents total shutdown** - Ensures critical services continue

### Basic Pod Disruption Budget Example
```yaml
# emergency-procedures.yaml
# Web service - keep at least 2 running
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-service-pdb
  namespace: production
spec:
  minAvailable: 2             # Always keep at least 2 pods running
  selector:
    matchLabels:
      app: web-service
---
# API service - keep at least 80% running
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-service-pdb
  namespace: production
spec:
  minAvailable: 80%           # Keep at least 80% of pods running
  selector:
    matchLabels:
      app: api-service
---
# Database - allow maximum 1 to be down
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: database-pdb
  namespace: production
spec:
  maxUnavailable: 1           # Allow maximum 1 pod to be unavailable
  selector:
    matchLabels:
      app: database
---
# Background jobs - can all be down temporarily
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: background-jobs-pdb
  namespace: production
spec:
  maxUnavailable: 100%        # All can be down during maintenance
  selector:
    matchLabels:
      app: background-jobs
```

### Step-by-Step: Set Emergency Procedures
```bash
# Step 1: Create deployments to test with
kubectl create deployment web-service --image=nginx --replicas=5 -n production
kubectl label deployment web-service app=web-service -n production

kubectl create deployment api-service --image=nginx --replicas=10 -n production
kubectl label deployment api-service app=api-service -n production

kubectl create deployment database --image=postgres --replicas=3 -n production
kubectl label deployment database app=database -n production

# Step 2: Apply pod disruption budgets
kubectl apply -f emergency-procedures.yaml

# Step 3: Check PDBs
kubectl get pdb -n production

# Expected output:
NAME               MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
web-service-pdb    2               N/A               3                     1m
api-service-pdb    80%             N/A               2                     1m
database-pdb       N/A             1                 1                     1m

# Step 4: Test disruption budget - try to drain a node
kubectl get nodes
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# PDB will prevent draining if it would violate the budget!

# Step 5: Check PDB status
kubectl describe pdb web-service-pdb -n production
```

## 5. Admission Controllers ✅ (Quality Inspectors)

### What are Admission Controllers?
**Quality inspectors** - check every request before it's approved

**Real-Life Example:** Building permit office
- **Safety Inspector** - Checks if building meets safety codes
- **Zoning Inspector** - Checks if building fits neighborhood rules
- **Budget Inspector** - Checks if you have enough money
- **Code Inspector** - Checks if plans follow building codes

### Built-in Admission Controllers
```
✅ KUBERNETES QUALITY INSPECTORS

🔐 PodSecurity
├── Checks pod security standards
├── Enforces security policies
└── Prevents dangerous configurations

💰 ResourceQuota
├── Checks namespace resource limits
├── Prevents exceeding budgets
└── Tracks resource usage

🏗️ LimitRanger
├── Checks individual resource limits
├── Applies default values
└── Enforces min/max constraints

🔍 NamespaceLifecycle
├── Prevents using terminating namespaces
├── Ensures namespace exists
└── Handles namespace cleanup

🏷️ DefaultStorageClass
├── Assigns default storage class
├── Ensures PVCs have storage class
└── Prevents storage issues
```

### Custom Admission Controller Example (Webhook)
```yaml
# custom-inspector.yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionWebhook
metadata:
  name: pizza-policy-webhook
webhooks:
  - name: pizza.policy.webhook
    clientConfig:
      service:
        name: pizza-policy-service
        namespace: kube-system
        path: "/validate"
    rules:
      - operations: ["CREATE", "UPDATE"]
        apiGroups: ["apps"]
        apiVersions: ["v1"]
        resources: ["deployments"]
    admissionReviewVersions: ["v1", "v1beta1"]
    sideEffects: None
    failurePolicy: Fail
```

### Policy Enforcement Tools

#### Open Policy Agent (OPA) Gatekeeper
```yaml
# gatekeeper-policies.yaml
# Constraint Template
apiVersion: templates.gatekeeper.sh/v1beta1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        properties:
          labels:
            type: array
            items:
              type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels
        
        violation[{"msg": msg}] {
          required := input.parameters.labels
          provided := input.review.object.metadata.labels
          missing := required[_]
          not provided[missing]
          msg := sprintf("Missing required label: %v", [missing])
        }
---
# Constraint
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: must-have-owner
spec:
  match:
    kinds:
      - apiGroups: ["apps"]
        kinds: ["Deployment"]
    namespaces: ["production"]
  parameters:
    labels: ["owner", "environment", "cost-center"]
```

## 7. Policy Best Practices 🌟

### 1. Environment-Based Policies
- **Development** - Relaxed policies for experimentation
- **Staging** - Moderate policies for testing
- **Production** - Strict policies for stability

### 2. Resource Management
- Set appropriate quotas based on team needs
- Use limit ranges to prevent resource abuse
- Monitor quota usage regularly

### 3. Availability Planning
- Use PDBs for critical services
- Plan for maintenance windows
- Test disruption scenarios

### 4. Security Policies
- Implement network policies in production
- Use pod security standards
- Regular policy audits

### 5. Monitoring and Alerting
- Monitor policy violations
- Set up alerts for quota exhaustion
- Track policy effectiveness

## 8. Policy Troubleshooting 🔧

### Common Issues and Solutions

#### 1. Resource Quota Exceeded
```bash
kubectl describe resourcequota <quota-name> -n <namespace>
# Check current usage vs limits
```

#### 2. Pod Creation Blocked by Limit Range
```bash
kubectl describe limitrange <limitrange-name> -n <namespace>
# Check if pod resources are within limits
```

#### 3. PDB Preventing Node Drain
```bash
kubectl get pdb -A
kubectl describe pdb <pdb-name>
# Check if enough pods are available
```

#### 4. Network Policy Blocking Traffic
```bash
kubectl describe networkpolicy <policy-name>
# Check if traffic is allowed by policies
```
