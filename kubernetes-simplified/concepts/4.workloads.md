# Kubernetes Workloads 🏗️

## What are Workloads? (Simple Explanation)

Think of workloads like **different types of businesses** you can run in your city!

### Real-Life Example 🏪
In a city, you have different types of businesses:
- **Restaurant (Pod)** - Single business serving customers
- **Restaurant Chain (Deployment)** - Multiple identical restaurants
- **Bank Branch (StatefulSet)** - Each branch has unique identity and records
- **Newspaper Delivery (DaemonSet)** - One service per neighborhood
- **Construction Project (Job)** - Work until project is complete
- **Street Cleaning (CronJob)** - Scheduled recurring work

**Kubernetes workloads work the same way!**

## 1. Pod 🏠 (Single Restaurant)

### What is a Pod?
**Simplest workload** - like a single restaurant serving customers

**Real-Life Example:** A small pizza shop
- One building (pod)
- Pizza chef (main container)
- Maybe a cashier helper (sidecar container)
- Shared utilities (network, storage)

### Basic Pod Example
```yaml
# simple-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-shop
  labels:
    app: pizza
spec:
  containers:
    - name: pizza-chef
      image: nginx:latest
      ports:
        - containerPort: 80
      resources:
        requests:
          memory: "64Mi"
          cpu: "250m"
        limits:
          memory: "128Mi"
          cpu: "500m"
```

### Step-by-Step: Create Your First Pod
```bash
# Step 1: Create the pod
kubectl apply -f simple-pod.yaml

# Step 2: Check if it's running
kubectl get pods

# Expected output:
NAME         READY   STATUS    RESTARTS   AGE
pizza-shop   1/1     Running   0          30s

# Step 3: See detailed information
kubectl describe pod pizza-shop

# Step 4: Check logs
kubectl logs pizza-shop

# Step 5: Access the pod (for testing)
kubectl port-forward pod/pizza-shop 8080:80
# Now visit http://localhost:8080
```

### Multi-Container Pod (Pizza Shop + Cashier)
```yaml
# multi-container-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-shop-with-cashier
spec:
  containers:
    - name: pizza-chef          # Main application
      image: nginx:latest
      ports:
        - containerPort: 80
    - name: cashier             # Helper (sidecar)
      image: busybox:latest
      command: ['sh', '-c', 'while true; do echo "Taking orders..."; sleep 30; done']
```

### Advanced Pod Concepts

#### Init Containers (Setup Workers)
```yaml
# pod-with-init.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-shop-with-setup
spec:
  initContainers:
  - name: setup-database
    image: busybox:latest
    command: ['sh', '-c', 'echo "Setting up database..."; sleep 10; echo "Database ready!"']
  - name: download-assets
    image: busybox:latest
    command: ['sh', '-c', 'echo "Downloading assets..."; sleep 5; echo "Assets ready!"']
  containers:
  - name: pizza-chef
    image: nginx:latest
    ports:
    - containerPort: 80
```

**Init Container Features:**
- Run sequentially before main containers
- Must complete successfully
- Perfect for setup tasks, migrations, downloads
- Share volumes with main containers

#### Sidecar Containers (Helper Workers)
```yaml
# pod-with-sidecar.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-shop-with-helpers
spec:
  containers:
  - name: pizza-chef          # Main application
    image: nginx:latest
    ports:
    - containerPort: 80
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/nginx
  - name: log-shipper         # Sidecar for log processing
    image: fluentd:latest
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/nginx
  - name: metrics-collector   # Sidecar for metrics
    image: prometheus/node-exporter:latest
    ports:
    - containerPort: 9100
  volumes:
  - name: shared-logs
    emptyDir: {}
```

**Common Sidecar Patterns:**
- **Log Shipping** - Collect and forward logs
- **Metrics Collection** - Gather application metrics
- **Service Mesh Proxy** - Handle network traffic (Istio, Linkerd)
- **Configuration Reload** - Watch and reload config changes
- **Security Scanning** - Monitor for vulnerabilities

#### Ephemeral Containers (Debug Helpers)
```bash
# Add temporary debug container to running pod
kubectl debug pizza-shop -it --image=busybox --target=pizza-chef

# Debug with different image
kubectl debug pizza-shop -it --image=ubuntu --share-processes

# Copy pod for debugging
kubectl debug pizza-shop -it --copy-to=pizza-shop-debug --container=debugger --image=busybox
```

**Ephemeral Container Features:**
- Temporary containers for debugging
- Don't restart with pod
- Can share process namespace
- Perfect for troubleshooting production issues

## 2. Deployment 🏪 (Restaurant Chain)

### What is a Deployment?
**Most common workload** - like managing a restaurant chain

**Real-Life Example:** McDonald's franchise
- Multiple identical restaurants (pods)
- If one closes, open another
- Can expand to more locations
- All serve the same menu (same container image)

### Basic Deployment Example
```yaml
# pizza-chain.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pizza-chain
  labels:
    app: pizza
spec:
  replicas: 3                    # I want 3 pizza shops
  selector:
    matchLabels:
      app: pizza
  template:                      # Template for each pizza shop
    metadata:
      labels:
        app: pizza
    spec:
      containers:
      - name: pizza-chef
        image: nginx:latest
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "250m"
```

### Step-by-Step: Deploy Your Restaurant Chain
```bash
# Step 1: Create the deployment
kubectl apply -f pizza-chain.yaml

# Step 2: Check deployment status
kubectl get deployments

# Expected output:
NAME          READY   UP-TO-DATE   AVAILABLE   AGE
pizza-chain   3/3     3            3           1m

# Step 3: See the pods created
kubectl get pods -l app=pizza

# Expected output:
NAME                           READY   STATUS    RESTARTS   AGE
pizza-chain-7d4b8c8f9d-abc12   1/1     Running   0          1m
pizza-chain-7d4b8c8f9d-def34   1/1     Running   0          1m
pizza-chain-7d4b8c8f9d-ghi56   1/1     Running   0          1m

# Step 4: Scale your chain (add more restaurants)
kubectl scale deployment pizza-chain --replicas=5

# Step 5: Check scaling
kubectl get pods -l app=pizza
# Now you'll see 5 pods!
```

### Deployment Features

#### 1. Self-Healing 🔧
```bash
# Delete a pod (simulate restaurant closure)
kubectl delete pod pizza-chain-7d4b8c8f9d-abc12

# Immediately check pods
kubectl get pods -l app=pizza
# You'll see a new pod being created automatically!
```

#### 2. Rolling Updates 🔄
```yaml
# Update your pizza recipe (new image version)
kubectl set image deployment/pizza-chain pizza-chef=nginx:1.21

# Watch the rolling update
kubectl rollout status deployment/pizza-chain

# What happens:
# 1. Creates new pod with new image
# 2. Waits for it to be ready
# 3. Terminates old pod
# 4. Repeats until all pods are updated
```

**Rolling Update Strategies:**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pizza-chain
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1      # Max pods that can be unavailable
      maxSurge: 1           # Max extra pods during update
  progressDeadlineSeconds: 600  # Timeout for rollout
```

**Recreate Strategy (Alternative):**
```yaml
spec:
  strategy:
    type: Recreate          # Kill all pods, then create new ones
    # Causes downtime but ensures no mixed versions
```

#### 3. Rollback 🔙
```bash
# Oops! New recipe is bad, rollback!
kubectl rollout undo deployment/pizza-chain

# Rollback to specific revision
kubectl rollout undo deployment/pizza-chain --to-revision=2

# Check rollout history
kubectl rollout history deployment/pizza-chain

# Pause rollout (stop in middle)
kubectl rollout pause deployment/pizza-chain

# Resume rollout
kubectl rollout resume deployment/pizza-chain
```

#### 4. Advanced Deployment Patterns

**Blue-Green Deployment:**
```bash
# Create new version (green)
kubectl create deployment pizza-green --image=nginx:1.21
kubectl scale deployment pizza-green --replicas=3

# Switch service to green
kubectl patch service pizza-service -p '{"spec":{"selector":{"version":"green"}}}'

# Remove old version (blue)
kubectl delete deployment pizza-blue
```

**Canary Deployment:**
```yaml
# Deploy small percentage of new version
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pizza-canary
spec:
  replicas: 1              # Only 1 pod (10% of traffic)
  selector:
    matchLabels:
      app: pizza
      version: canary
  template:
    metadata:
      labels:
        app: pizza
        version: canary
    spec:
      containers:
      - name: pizza-chef
        image: nginx:1.21     # New version
```

**ReplicaSet (Behind the Scenes):**
- Deployments create and manage ReplicaSets
- ReplicaSets ensure desired number of pod replicas
- Each deployment revision creates a new ReplicaSet
- Old ReplicaSets kept for rollback (configurable)

```bash
# See ReplicaSets created by deployment
kubectl get replicasets

# Expected output:
NAME                     DESIRED   CURRENT   READY   AGE
pizza-chain-7d4b8c8f9d   3         3         3       5m
pizza-chain-6c5b7d8e9f   0         0         0       2m  # Old version
```

## 3. StatefulSet 🏦 (Bank Branches)

### What is a StatefulSet?
**For apps that need identity** - like bank branches with unique records

**Real-Life Example:** Bank branches
- Each branch has unique identity (Bank-0, Bank-1, Bank-2)
- Each has its own customer records (persistent storage)
- Main branch must start first
- If branch closes, it reopens with same identity and records

### StatefulSet Example
```yaml
# bank-branches.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: bank-branches
spec:
  serviceName: "bank-service"
  replicas: 3
  selector:
    matchLabels:
      app: bank
  template:
    metadata:
      labels:
        app: bank
    spec:
      containers:
      - name: bank-app
        image: nginx:latest
        ports:
        - containerPort: 80
        volumeMounts:
        - name: bank-records
          mountPath: /data
  volumeClaimTemplates:           # Each branch gets its own storage
  - metadata:
      name: bank-records
    spec:
      accessModes: [ "ReadWriteOnce" ]
      resources:
        requests:
          storage: 1Gi
```

### Step-by-Step: Open Bank Branches
```bash
# Step 1: Create StatefulSet
kubectl apply -f bank-branches.yaml

# Step 2: Watch pods start in order
kubectl get pods -l app=bank -w

# You'll see:
# bank-branches-0   1/1   Running   (starts first)
# bank-branches-1   1/1   Running   (starts after 0 is ready)
# bank-branches-2   1/1   Running   (starts after 1 is ready)

# Step 3: Check persistent volumes
kubectl get pvc
# Each pod gets its own storage:
# bank-records-bank-branches-0
# bank-records-bank-branches-1  
# bank-records-bank-branches-2
```

### StatefulSet vs Deployment

| Feature | Deployment 🏪 | StatefulSet 🏦 |
|---------|---------------|----------------|
| **Pod Names** | Random (pizza-chain-abc123) | Ordered (bank-branches-0) |
| **Storage** | Shared or none | Each pod gets own storage |
| **Startup** | All at once | One by one, in order |
| **Use Case** | Stateless apps (web servers) | Databases, message queues |

## 4. DaemonSet 📰 (Newspaper Delivery)

### What is a DaemonSet?
**One pod per node** - like newspaper delivery service

**Real-Life Example:** Newspaper delivery
- One delivery person per neighborhood (node)
- Covers every area of the city
- If new neighborhood added, assign delivery person
- If neighborhood removed, delivery person leaves

### DaemonSet Example
```yaml
# newspaper-delivery.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: newspaper-delivery
spec:
  selector:
    matchLabels:
      app: newspaper
  template:
    metadata:
      labels:
        app: newspaper
    spec:
      containers:
      - name: delivery-person
        image: busybox:latest
        command: ['sh', '-c', 'while true; do echo "Delivering newspapers on $(hostname)"; sleep 60; done']
        resources:
          requests:
            memory: "32Mi"
            cpu: "100m"
```

### Step-by-Step: Start Newspaper Delivery
```bash
# Step 1: Create DaemonSet
kubectl apply -f newspaper-delivery.yaml

# Step 2: Check pods on each node
kubectl get pods -l app=newspaper -o wide

# Expected output (if you have 3 nodes):
NAME                       READY   STATUS    RESTARTS   AGE   NODE
newspaper-delivery-abc12   1/1     Running   0          1m    worker-node-1
newspaper-delivery-def34   1/1     Running   0          1m    worker-node-2
newspaper-delivery-ghi56   1/1     Running   0          1m    worker-node-3

# Step 3: Check logs from any delivery person
kubectl logs newspaper-delivery-abc12
```

### Common DaemonSet Use Cases
- **Log Collection** (Fluentd, Filebeat)
- **Monitoring** (Node Exporter, Datadog agent)
- **Network** (Calico, Flannel CNI)
- **Security** (Falco security monitoring)

## 5. Job 🚧 (Construction Project)

### What is a Job?
**Run until completion** - like a construction project

**Real-Life Example:** Building a house
- Has specific goal (build the house)
- Works until finished
- Doesn't run forever
- If worker gets sick, hire replacement
- When done, project is complete

### Job Example
```yaml
# house-construction.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: house-construction
spec:
  completions: 1              # Need 1 successful completion
  parallelism: 1              # Use 1 worker at a time
  backoffLimit: 3             # Retry up to 3 times if failed
  template:
    spec:
      restartPolicy: Never    # Don't restart on failure
      containers:
      - name: construction-worker
        image: busybox:latest
        command: ['sh', '-c', 'echo "Starting construction..."; sleep 30; echo "House built successfully!"; exit 0']
```

### Step-by-Step: Build a House
```bash
# Step 1: Start construction project
kubectl apply -f house-construction.yaml

# Step 2: Watch the job progress
kubectl get jobs -w

# You'll see:
NAME                COMPLETIONS   DURATION   AGE
house-construction  0/1           0s         5s
house-construction  1/1           35s        35s  ← Job completed!

# Step 3: Check the worker pod
kubectl get pods -l job-name=house-construction

# Step 4: See construction logs
kubectl logs job/house-construction
```

### Job Types & Patterns

#### Parallel Jobs (Multiple Workers)
```yaml
# parallel-construction.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: parallel-construction
spec:
  completions: 6              # Need 6 tasks completed
  parallelism: 3              # Use 3 workers simultaneously
  backoffLimit: 2             # Retry failed jobs 2 times
  activeDeadlineSeconds: 300  # Timeout after 5 minutes
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: worker
        image: busybox:latest
        command: ['sh', '-c', 'echo "Worker $(hostname) building room..."; sleep 20; echo "Room completed!"']
```

#### Work Queue Pattern
```yaml
# work-queue-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: work-queue-processor
spec:
  parallelism: 5              # 5 workers
  completions: 1              # Keep running until queue is empty
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: worker
        image: work-queue-processor:latest
        env:
        - name: QUEUE_URL
          value: "redis://queue-service:6379"
        command: ['python', 'process_queue.py']
```

#### Indexed Jobs (Kubernetes 1.21+)
```yaml
# indexed-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: indexed-processing
spec:
  completions: 10
  parallelism: 3
  completionMode: Indexed     # Each pod gets unique index
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: processor
        image: data-processor:latest
        env:
        - name: JOB_COMPLETION_INDEX  # Automatic env var with index
          valueFrom:
            fieldRef:
              fieldPath: metadata.annotations['batch.kubernetes.io/job-completion-index']
```

## 6. CronJob 🧹 (Street Cleaning)

### What is a CronJob?
**Scheduled recurring work** - like street cleaning service

**Real-Life Example:** Street cleaning
- Happens every night at 2 AM
- Runs automatically on schedule
- Cleans up and finishes
- Repeats next night

### CronJob Example
```yaml
# street-cleaning.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: street-cleaning
spec:
  schedule: "0 2 * * *"        # Every day at 2 AM (cron format)
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: cleaner
            image: busybox:latest
            command: ['sh', '-c', 'echo "Cleaning streets at $(date)"; sleep 10; echo "Streets cleaned!"']
```

### Cron Schedule Format
```
# ┌───────────── minute (0 - 59)
# │ ┌───────────── hour (0 - 23)
# │ │ ┌───────────── day of month (1 - 31)
# │ │ │ ┌───────────── month (1 - 12)
# │ │ │ │ ┌───────────── day of week (0 - 6) (Sunday to Saturday)
# │ │ │ │ │
# * * * * *

"0 2 * * *"     # Every day at 2:00 AM
"*/15 * * * *"  # Every 15 minutes
"0 0 * * 0"     # Every Sunday at midnight
"0 9 * * 1-5"   # Every weekday at 9 AM
```

### Step-by-Step: Schedule Street Cleaning
```bash
# Step 1: Create CronJob
kubectl apply -f street-cleaning.yaml

# Step 2: Check CronJob status
kubectl get cronjobs

# Expected output:
NAME             SCHEDULE    SUSPEND   ACTIVE   LAST SCHEDULE   AGE
street-cleaning  0 2 * * *   False     0        <none>          1m

# Step 3: Manually trigger for testing (don't wait until 2 AM!)
kubectl create job --from=cronjob/street-cleaning manual-cleaning

# Step 4: Check the manual job
kubectl get jobs
kubectl logs job/manual-cleaning
```

## Workload Comparison Table 📊

| Workload | Use Case | Example | Restart Policy | Storage |
|----------|----------|---------|----------------|---------|
| **Pod** 🏠 | Single instance | Development/Testing | Always | Temporary |
| **Deployment** 🏪 | Stateless apps | Web servers, APIs | Always | Shared/None |
| **StatefulSet** 🏦 | Stateful apps | Databases, Queues | Always | Per-pod persistent |
| **DaemonSet** 📰 | Node services | Logging, Monitoring | Always | Node-local |
| **Job** 🚧 | Run-to-completion | Data processing | Never/OnFailure | Temporary |
| **CronJob** 🧹 | Scheduled tasks | Backups, Cleanup | Never/OnFailure | Temporary |

## Choosing the Right Workload 🤔

### Decision Tree
```
🤔 What type of application do you have?

├── 🌐 Web Application (serves users continuously)
│   ├── 💾 Stores data/state? → StatefulSet 🏦
│   └── 🔄 Stateless? → Deployment 🏪
│
├── 🔧 System Service (runs on every node)
│   └── DaemonSet 📰
│
├── 📊 Data Processing (runs once and finishes)
│   ├── 🕐 Scheduled regularly? → CronJob 🧹
│   └── 🚀 Run once? → Job 🚧
│
└── 🧪 Testing/Development
    └── Pod 🏠
```

## Hands-On Exercise: Complete Example 🎯

Let's create a complete web application with database:

### Step 1: Database (StatefulSet)
```yaml
# database.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql-db
spec:
  serviceName: mysql-service
  replicas: 1
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        env:
        - name: MYSQL_ROOT_PASSWORD
          value: "password123"
        - name: MYSQL_DATABASE
          value: "myapp"
        ports:
        - containerPort: 3306
        volumeMounts:
        - name: mysql-storage
          mountPath: /var/lib/mysql
  volumeClaimTemplates:
  - metadata:
      name: mysql-storage
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 1Gi
```

### Step 2: Web Application (Deployment)
```yaml
# webapp.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
      - name: webapp
        image: nginx:latest
        ports:
        - containerPort: 80
        env:
        - name: DB_HOST
          value: "mysql-service"
```

### Step 3: Database Backup (CronJob)
```yaml
# backup.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup
spec:
  schedule: "0 2 * * *"  # Every day at 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: backup
            image: mysql:8.0
            command: ['sh', '-c', 'mysqldump -h mysql-service -u root -ppassword123 myapp > /backup/backup-$(date +%Y%m%d).sql']
```

### Deploy Everything
```bash
# Deploy in order
kubectl apply -f database.yaml
kubectl apply -f webapp.yaml  
kubectl apply -f backup.yaml

# Check everything is running
kubectl get all
```

## Troubleshooting Workloads 🔧

### Common Issues and Solutions

#### 1. Pod Stuck in Pending
```bash
kubectl describe pod <pod-name>
```
**Common causes:**
- Insufficient resources
- No available nodes
- Volume mounting issues

#### 2. Pod CrashLoopBackOff
```bash
kubectl logs <pod-name>
kubectl logs <pod-name> --previous  # Previous crash logs
```
**Common causes:**
- Application errors
- Wrong configuration
- Missing dependencies

#### 3. Job Never Completes
```bash
kubectl describe job <job-name>
kubectl logs job/<job-name>
```
**Common causes:**
- Wrong restart policy
- Application doesn't exit
- Resource limits too low
