# Kubernetes Storage 💾

## What is Kubernetes Storage? (Simple Explanation)

Think of Kubernetes storage like **different types of storage in a city**!

### Real-Life Example 🏙️
In a city, you have different storage needs:
- **Temporary Storage (emptyDir)** - Like a desk drawer, gone when you leave office
- **Personal Storage (hostPath)** - Like your home closet, tied to your house
- **Bank Vault (Persistent Volume)** - Professional storage that survives everything
- **Storage Rental (Persistent Volume Claim)** - Renting space in the vault
- **Storage Companies (Storage Classes)** - Different types of storage services
- **File Cabinets (ConfigMaps/Secrets)** - Store documents and important papers

**Kubernetes storage works the same way!**

## 1. Temporary Storage - emptyDir 📝 (Desk Drawer)

### What is emptyDir?
**Temporary storage** - like a desk drawer that gets emptied when you leave the office

**Real-Life Example:** Office desk drawer
- Available while you're working (pod running)
- Shared with your office mate (containers in same pod)
- Gets emptied when you leave (pod restarts)
- Perfect for temporary files

### emptyDir Example
```yaml
# temp-storage-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: office-worker
spec:
  containers:
    - name: worker
      image: busybox:latest
      command: ['sh', '-c', 'echo "Working..." > /tmp/work.txt; sleep 3600']
      volumeMounts:
        - name: desk-drawer
          mountPath: /tmp
    - name: helper
      image: busybox:latest
      command: ['sh', '-c', 'while true; do ls -la /shared; sleep 30; done']
      volumeMounts:
        - name: desk-drawer
          mountPath: /shared
  volumes:
    - name: desk-drawer
      emptyDir: {}              # Temporary storage
```

### Step-by-Step: Use Desk Drawer
```bash
# Step 1: Create pod with shared temporary storage
kubectl apply -f temp-storage-pod.yaml

# Step 2: Check what worker created
kubectl exec office-worker -c worker -- ls -la /tmp

# Step 3: Check if helper can see the same files
kubectl exec office-worker -c helper -- ls -la /shared

# Step 4: Create file from helper
kubectl exec office-worker -c helper -- touch /shared/helper-file.txt

# Step 5: Check if worker can see it
kubectl exec office-worker -c worker -- ls -la /tmp

# Step 6: Restart pod (simulate leaving office)
kubectl delete pod office-worker
kubectl apply -f temp-storage-pod.yaml

# Step 7: Check if files are gone
kubectl exec office-worker -c worker -- ls -la /tmp
# Files are gone! 📝➡️🗑️
```

### emptyDir Use Cases
- **Temporary files** - Processing data, cache files
- **Shared space** - Communication between containers in same pod
- **Scratch space** - Temporary downloads, builds

## 2. Node Storage - hostPath 🏠 (Home Closet)

### What is hostPath?
**Node-specific storage** - like your home closet, tied to your house

**Real-Life Example:** Home closet
- Belongs to your specific house (node)
- Survives when you leave and come back (pod restarts)
- But if you move houses (pod moves to different node), you lose access
- Good for node-specific data

### hostPath Example
```yaml
# home-storage-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: homeowner
spec:
  containers:
  - name: resident
    image: busybox:latest
    command: ['sh', '-c', 'echo "My stuff" > /home-closet/belongings.txt; sleep 3600']
    volumeMounts:
    - name: home-closet
      mountPath: /home-closet
  volumes:
  - name: home-closet
    hostPath:
      path: /tmp/node-storage    # Path on the node
      type: DirectoryOrCreate    # Create if doesn't exist
```

### Step-by-Step: Use Home Closet
```bash
# Step 1: Create pod with node storage
kubectl apply -f home-storage-pod.yaml

# Step 2: Store something in the closet
kubectl exec homeowner -- echo "Important documents" > /home-closet/documents.txt

# Step 3: Check what's stored
kubectl exec homeowner -- ls -la /home-closet

# Step 4: Delete pod (simulate leaving home)
kubectl delete pod homeowner

# Step 5: Create new pod on same node
kubectl apply -f home-storage-pod.yaml

# Step 6: Check if stuff is still there
kubectl exec homeowner -- ls -la /home-closet
# Files are still there! 🏠✅

# Step 7: Check on the actual node
kubectl get pods -o wide  # See which node
# SSH to that node and check /tmp/node-storage
```

### hostPath Use Cases
- **Log files** - Node-specific logs
- **System monitoring** - Access node's system files
- **Development** - Quick testing with local files
- **DaemonSets** - Node agents that need node access

### ⚠️ hostPath Warnings
- **Not portable** - Pod tied to specific node
- **Security risk** - Can access node's file system
- **Not recommended for production** - Use Persistent Volumes instead

## 3. Persistent Volumes (PV) 🏦 (Bank Vault)

### What is a Persistent Volume?
**Professional storage service** - like a bank vault that survives everything

**Real-Life Example:** Bank vault
- Professional storage facility
- Survives disasters, moves, changes
- Has different types (safety deposit boxes, large vaults)
- Managed by professionals (storage admin)
- You rent space through agreements (PVC)

### Persistent Volume Lifecycle
```
🏦 BANK VAULT LIFECYCLE

1. 🏗️ PROVISIONING (Bank builds vault)
   ├── Static: Admin creates vault manually
   └── Dynamic: Vault created automatically when needed

2. 📋 BINDING (Customer rents vault space)
   ├── Customer submits rental application (PVC)
   ├── Bank finds suitable vault (PV)
   └── Signs rental agreement (Binding)

3. 💼 USING (Customer stores valuables)
   ├── Customer accesses vault with key
   ├── Stores and retrieves items
   └── Vault space is reserved for customer

4. 🔄 RECLAIMING (Rental ends)
   ├── Retain: Keep contents, vault stays reserved
   ├── Delete: Destroy contents and vault
   └── Recycle: Clean vault for next customer (deprecated)
```

### Static Persistent Volume Example
```yaml
# bank-vault.yaml (Created by storage admin)
apiVersion: v1
kind: PersistentVolume
metadata:
  name: bank-vault-1
spec:
  capacity:
    storage: 10Gi              # Vault size
  accessModes:
  - ReadWriteOnce              # One customer at a time
  persistentVolumeReclaimPolicy: Retain  # Keep contents when rental ends
  storageClassName: premium-ssd
  hostPath:                    # For demo - use real storage in production
    path: /mnt/vault-1
```

### Persistent Volume Claim (PVC) - Rental Agreement
```yaml
# vault-rental.yaml (Created by customer)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-vault-rental
spec:
  accessModes:
  - ReadWriteOnce              # I need exclusive access
  resources:
    requests:
      storage: 5Gi             # I need at least 5GB
  storageClassName: premium-ssd # I want premium storage
```

### Using PVC in Pod
```yaml
# vault-user.yaml
apiVersion: v1
kind: Pod
metadata:
  name: vault-customer
spec:
  containers:
  - name: customer
    image: busybox:latest
    command: ['sh', '-c', 'echo "Valuable data" > /my-vault/treasure.txt; sleep 3600']
    volumeMounts:
    - name: my-vault
      mountPath: /my-vault
  volumes:
  - name: my-vault
    persistentVolumeClaim:
      claimName: my-vault-rental  # Reference to rental agreement
```

### Step-by-Step: Rent Bank Vault
```bash
# Step 1: Storage admin creates vault
kubectl apply -f bank-vault.yaml

# Step 2: Check available vaults
kubectl get pv

# Expected output:
NAME           CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS      CLAIM   STORAGECLASS   AGE
bank-vault-1   10Gi       RWO            Retain           Available           premium-ssd    1m

# Step 3: Customer requests vault rental
kubectl apply -f vault-rental.yaml

# Step 4: Check rental status
kubectl get pvc

# Expected output:
NAME              STATUS   VOLUME         CAPACITY   ACCESS MODES   STORAGECLASS   AGE
my-vault-rental   Bound    bank-vault-1   10Gi       RWO            premium-ssd    30s

# Step 5: Customer uses vault
kubectl apply -f vault-user.yaml

# Step 6: Store valuables
kubectl exec vault-customer -- echo "Secret documents" > /my-vault/secrets.txt

# Step 7: Check contents
kubectl exec vault-customer -- ls -la /my-vault

# Step 8: Delete pod (customer leaves)
kubectl delete pod vault-customer

# Step 9: Create new pod (customer returns)
kubectl apply -f vault-user.yaml

# Step 10: Check if valuables are safe
kubectl exec vault-customer -- ls -la /my-vault
# Files are still there! 🏦✅
```

## 4. Storage Classes 🏪 (Storage Companies)

### What are Storage Classes?
**Different storage services** - like different storage companies with various offerings

**Real-Life Example:** Storage companies
- **Premium Storage Co.** - Fast SSD, expensive, high performance
- **Budget Storage Co.** - Slow HDD, cheap, basic service
- **Cloud Storage Co.** - Scalable, managed, pay-as-you-use
- **Local Storage Co.** - Fast access, but tied to location

### Container Storage Interface (CSI)
**Modern storage architecture** - standardized storage drivers

```
💾 CSI ARCHITECTURE

💻 Kubernetes API Server
        │
🔌 CSI Controller Plugin
        │
┌───────┼───────┐
│       │       │
💾 AWS EBS  💾 GCE PD  💾 Azure Disk
💾 Ceph     💾 NFS     💾 Local SSD
```

**CSI Benefits:**
- **Standardized Interface** - Same API for all storage types
- **Pluggable Architecture** - Easy to add new storage systems
- **Vendor Independence** - Not tied to specific cloud provider
- **Advanced Features** - Snapshots, cloning, expansion

### Storage Class Parameters by Provider

#### AWS EBS Storage Classes
```yaml
# aws-storage-classes.yaml
# High Performance SSD
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: aws-gp3-ssd
provisioner: ebs.csi.aws.com
allowVolumeExpansion: true
parameters:
  type: gp3
  iops: "3000"               # IOPS performance
  throughput: "125"          # MB/s throughput
  encrypted: "true"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
---
# Budget HDD Storage
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: aws-sc1-hdd
provisioner: ebs.csi.aws.com
parameters:
  type: sc1                  # Cold HDD
  encrypted: "false"
reclaimPolicy: Delete
```

#### Volume Snapshots (CSI Feature)
```yaml
# volume-snapshot-class.yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: csi-snapclass
deletionPolicy: Delete
driver: ebs.csi.aws.com
---
# volume-snapshot.yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: vault-backup-snapshot
spec:
  volumeSnapshotClassName: csi-snapclass
  source:
    persistentVolumeClaimName: my-vault-rental
```

### Dynamic Provisioning with Storage Classes
```yaml
# auto-vault-rental.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: auto-vault
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
  storageClassName: premium-ssd  # Automatically creates PV with this class
```

### Step-by-Step: Use Storage Companies
```bash
# Step 1: Create storage companies
kubectl apply -f storage-companies.yaml

# Step 2: Check available storage companies
kubectl get storageclass

# Expected output:
NAME                    PROVISIONER                AGE
premium-ssd             kubernetes.io/host-path    1m
budget-hdd              kubernetes.io/host-path    1m
standard (default)      kubernetes.io/host-path    1m

# Step 3: Request automatic vault (dynamic provisioning)
kubectl apply -f auto-vault-rental.yaml

# Step 4: Check what happened
kubectl get pvc
kubectl get pv

# You'll see PV was automatically created! 🏪➡️🏦
```

## 5. Access Modes 🚪 (Vault Access Rules)

### Understanding Access Modes
**How many customers can use vault simultaneously**

```
🚪 VAULT ACCESS MODES

ReadWriteOnce (RWO) 🔐
├── One customer at a time
├── Can read and write
└── Most common for databases

ReadOnlyMany (ROX) 👥📖
├── Multiple customers can read
├── Nobody can write
└── Good for shared configuration

ReadWriteMany (RWX) 👥✏️
├── Multiple customers can read and write
├── Requires special storage (NFS, etc.)
└── Good for shared file systems

ReadWriteOncePod (RWOP) 🔒
├── One pod at a time (strictest)
├── Even stricter than RWO
└── Kubernetes 1.22+ feature
```

### Access Mode Examples
```yaml
# Different access modes for different needs
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: database-storage
spec:
  accessModes:
  - ReadWriteOnce          # Database needs exclusive access
  resources:
    requests:
      storage: 50Gi
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-config
spec:
  accessModes:
  - ReadOnlyMany           # Multiple pods can read config
  resources:
    requests:
      storage: 1Gi
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-files
spec:
  accessModes:
  - ReadWriteMany          # Multiple pods can read/write
  resources:
    requests:
      storage: 100Gi
```

## 6. Volume Types Comparison 📋

| Volume Type | Persistence | Scope | Use Case | Example |
|-------------|-------------|-------|----------|---------|
| **emptyDir** | Pod lifetime | Pod | Temporary files | Cache, scratch space |
| **hostPath** | Node lifetime | Node | Node-specific data | Logs, system files |
| **PV/PVC** | Independent | Cluster | Production data | Databases, user files |
| **configMap** | Independent | Namespace | Configuration | App settings, config files |
| **secret** | Independent | Namespace | Sensitive data | Passwords, certificates |

## 7. Real-World Example: Complete Database Setup 🎯

Let's create a complete MySQL database with persistent storage!

### Step 1: Storage Class (Storage Company)
```yaml
# mysql-storage-class.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: mysql-storage
provisioner: kubernetes.io/host-path
parameters:
  type: ssd
reclaimPolicy: Retain
allowVolumeExpansion: true
```

### Step 2: Persistent Volume Claim (Vault Rental)
```yaml
# mysql-pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-data
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
  storageClassName: mysql-storage
```

### Step 3: MySQL Database with Persistent Storage
```yaml
# mysql-database.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql-database
spec:
  serviceName: mysql-service
  replicas: 1
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        env:
        - name: MYSQL_ROOT_PASSWORD
          value: "secure-password-123"
        - name: MYSQL_DATABASE
          value: "myapp"
        - name: MYSQL_USER
          value: "appuser"
        - name: MYSQL_PASSWORD
          value: "user-password-456"
        ports:
        - containerPort: 3306
          name: mysql
        volumeMounts:
        - name: mysql-data
          mountPath: /var/lib/mysql    # MySQL data directory
        resources:
          requests:
            memory: "512Mi"
            cpu: "500m"
          limits:
            memory: "1Gi"
            cpu: "1000m"
      volumes:
      - name: mysql-data
        persistentVolumeClaim:
          claimName: mysql-data
---
apiVersion: v1
kind: Service
metadata:
  name: mysql-service
spec:
  selector:
    app: mysql
  ports:
  - port: 3306
    targetPort: 3306
  clusterIP: None  # Headless service for StatefulSet
```

### Step 4: Test Application
```yaml
# mysql-client.yaml
apiVersion: v1
kind: Pod
metadata:
  name: mysql-client
spec:
  containers:
    - name: client
      image: mysql:8.0
      command: ['sleep', '3600']
      env:
        - name: MYSQL_HOST
          value: "mysql-service"
        - name: MYSQL_USER
          value: "appuser"
        - name: MYSQL_PASSWORD
          value: "user-password-456"
        - name: MYSQL_DATABASE
          value: "myapp"
```

### Step 5: Deploy and Test
```bash
# Step 1: Create storage infrastructure
kubectl apply -f mysql-storage-class.yaml
kubectl apply -f mysql-pvc.yaml

# Step 2: Deploy MySQL database
kubectl apply -f mysql-database.yaml

# Step 3: Wait for database to be ready
kubectl wait --for=condition=ready pod -l app=mysql --timeout=300s

# Step 4: Create test client
kubectl apply -f mysql-client.yaml

# Step 5: Connect to database and create data
kubectl exec -it mysql-client -- mysql -h mysql-service -u appuser -puser-password-456 myapp

# Inside MySQL:
CREATE TABLE users (id INT PRIMARY KEY, name VARCHAR(50));
INSERT INTO users VALUES (1, 'Alice'), (2, 'Bob');
SELECT * FROM users;
exit

# Step 6: Test persistence - delete database pod
kubectl delete pod mysql-database-0

# Step 7: Wait for pod to restart
kubectl wait --for=condition=ready pod -l app=mysql --timeout=300s

# Step 8: Check if data survived
kubectl exec -it mysql-client -- mysql -h mysql-service -u appuser -puser-password-456 myapp -e "SELECT * FROM users;"

# Data is still there! 🏦✅
```

## 8. Storage Best Practices 🌟

### 1. Choose Right Storage Type
```bash
# For temporary data
emptyDir: {}

# For configuration
configMap:
  name: app-config

# For secrets
secret:
  secretName: app-secrets

# For persistent data
persistentVolumeClaim:
  claimName: app-data
```

### 2. Use Appropriate Access Modes
- **ReadWriteOnce** - Databases, single-writer applications
- **ReadOnlyMany** - Configuration files, static content
- **ReadWriteMany** - Shared file systems (requires special storage)

### 3. Set Resource Requests Properly
```yaml
resources:
  requests:
    storage: 10Gi    # Start with what you need
# Can expand later if storage class allows
```

### 4. Use Storage Classes
```yaml
# Production
storageClassName: premium-ssd

# Development  
storageClassName: standard

# Backup/Archive
storageClassName: slow-hdd
```

### 5. Monitor Storage Usage
```bash
# Check PVC usage
kubectl get pvc

# Check PV status
kubectl get pv

# Describe for details
kubectl describe pvc <pvc-name>
```

## 9. Troubleshooting Storage 🔧

### Common Issues and Solutions

#### 1. PVC Stuck in Pending
```bash
kubectl describe pvc <pvc-name>
```
**Common causes:**
- No available PV matches requirements
- Storage class doesn't exist
- Insufficient storage capacity

#### 2. Pod Can't Mount Volume
```bash
kubectl describe pod <pod-name>
```
**Common causes:**
- PVC not bound
- Access mode conflicts
- Node doesn't support volume type

#### 3. Data Loss After Pod Restart
```bash
# Check volume type
kubectl describe pod <pod-name>
```
**Common causes:**
- Using emptyDir instead of PVC
- PV reclaim policy set to Delete
- Volume not properly mounted

#### 4. Storage Full
```bash
# Check storage usage in pod
kubectl exec <pod-name> -- df -h

# Check PVC capacity
kubectl get pvc
```
