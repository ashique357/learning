# Containers

Think of containers like **standardized shipping containers** for your applications. Just as shipping containers revolutionized global trade by providing a standard way to package and transport goods, software containers revolutionize application deployment by providing a standard way to package and run applications.

## Understanding Containers

### The Container Analogy
Imagine you're **moving to a new city**:

**Traditional Way (Without Containers):**
- Pack everything loosely in boxes
- Hope nothing breaks during transport
- Spend days setting up utilities in new home
- Deal with compatibility issues (different electrical outlets, etc.)

**Container Way:**
- Everything packed in a standardized, self-contained unit
- Includes all utilities and dependencies
- Plug-and-play setup anywhere
- Guaranteed to work the same way everywhere

## What is a Container?

A container is like a **complete apartment unit** that includes:
- **Your application** (the tenant)
- **All dependencies** (furniture and appliances)
- **Runtime environment** (utilities and services)
- **Configuration** (personal settings)


## Container Images: The Blueprint

Container images are like **architectural blueprints** for creating containers.

### Dockerfile: Construction Instructions

```dockerfile
# Use official Node.js runtime as base (foundation)
FROM node:18-alpine

# Set working directory (choose apartment layout)
WORKDIR /app

# Copy package files (move in furniture list)
COPY package*.json ./

# Install dependencies (buy and install furniture)
RUN npm ci --only=production

# Copy application code (move in personal belongings)
COPY 03-containers .

# Create non-root user (assign apartment tenant)
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001

# Change ownership (transfer apartment ownership)
RUN chown -R nextjs:nodejs /app
USER nextjs

# Expose port (install doorbell)
EXPOSE 3000

# Health check (install smoke detector)
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# Start application (move in and start living)
CMD ["npm", "start"]
```

### Multi-Stage Builds: Efficient Construction
```dockerfile
# Stage 1: Build environment (construction site)
FROM node:18-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# Stage 2: Production environment (finished apartment)
FROM node:18-alpine AS production

WORKDIR /app

# Copy only production dependencies
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force

# Copy built application from builder stage
COPY --from=builder /app/dist ./dist

# Create non-root user
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001

RUN chown -R nextjs:nodejs /app
USER nextjs

EXPOSE 3000

CMD ["node", "dist/index.js"]
```

## Container Lifecycle

### Container States
Like **apartment occupancy states**:

```bash
# Container lifecycle
Created → Running → Paused → Stopped → Removed

# Check container status
docker ps                    # Running containers (occupied apartments)
docker ps -a                 # All containers (all apartments)
docker images               # Available images (blueprints)
```

### Container Operations
```bash
# Build image (construct apartment from blueprint)
docker build -t myapp:v1.0 .

# Run container (move tenant into apartment)
docker run -d --name myapp-container \
  -p 3000:3000 \
  -e NODE_ENV=production \
  -v /host/data:/app/data \
  myapp:v1.0

# Inspect container (apartment inspection)
docker inspect myapp-container

# View logs (check apartment activity)
docker logs myapp-container
docker logs -f myapp-container  # Follow logs

# Execute commands (visit apartment)
docker exec -it myapp-container /bin/sh

# Stop container (tenant moves out temporarily)
docker stop myapp-container

# Start container (tenant moves back in)
docker start myapp-container

# Remove container (demolish apartment)
docker rm myapp-container

# Remove image (destroy blueprint)
docker rmi myapp:v1.0
```

## Container Networking

### Network Types
Like **different communication systems** in a building:

#### Bridge Network (Default)
```bash
# Create custom bridge network (private building network)
docker network create --driver bridge myapp-network

# Run containers on custom network
docker run -d --name web-app --network myapp-network myapp:v1.0
docker run -d --name database --network myapp-network postgres:13

# Containers can communicate by name
# web-app can reach database at hostname "database"
```

#### Host Network
```bash
# Use host network (direct access to building utilities)
docker run -d --name myapp --network host myapp:v1.0
# Container uses host's network directly
```

#### None Network
```bash
# No network (isolated apartment)
docker run -d --name isolated-app --network none myapp:v1.0
# Container has no network access
```

### Port Mapping
```bash
# Map container port to host port (apartment doorbell to building entrance)
docker run -d -p 8080:3000 myapp:v1.0
# Host port 8080 → Container port 3000

# Map to specific host interface
docker run -d -p 127.0.0.1:8080:3000 myapp:v1.0
# Only localhost can access

# Map random host port
docker run -d -P myapp:v1.0
# Docker assigns random available port
```

## Container Storage

### Volume Types
Like **different storage solutions** for apartments:

#### Named Volumes (Building Storage Units)
```bash
# Create named volume (rent storage unit)
docker volume create myapp-data

# Use named volume
docker run -d -v myapp-data:/app/data myapp:v1.0

# List volumes
docker volume ls

# Inspect volume
docker volume inspect myapp-data

# Remove volume
docker volume rm myapp-data
```

#### Bind Mounts (Personal Storage)
```bash
# Mount host directory (use personal storage)
docker run -d -v /host/path:/container/path myapp:v1.0

# Mount with specific permissions
docker run -d -v /host/path:/container/path:ro myapp:v1.0  # Read-only
```

#### tmpfs Mounts (Temporary Storage)
```bash
# Mount temporary filesystem (temporary workspace)
docker run -d --tmpfs /tmp myapp:v1.0
# Data stored in memory, lost when container stops
```

## Container Security

### Security Best Practices
Like **apartment security measures**:

#### Run as Non-Root User
```dockerfile
# Create and use non-root user (don't give master keys to tenant)
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup

USER appuser
```

#### Read-Only Root Filesystem
```bash
# Make root filesystem read-only (apartment structure can't be modified)
docker run -d --read-only --tmpfs /tmp myapp:v1.0
```

#### Resource Limits
```bash
# Set resource limits (utility usage limits)
docker run -d \
  --memory=512m \
  --cpus=0.5 \
  --pids-limit=100 \
  myapp:v1.0
```

#### Security Options
```bash
# Drop capabilities (remove unnecessary permissions)
docker run -d \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --security-opt=no-new-privileges:true \
  myapp:v1.0
```

### Container Scanning
```bash
# Scan image for vulnerabilities (security inspection)
docker scan myapp:v1.0

# Use Trivy for comprehensive scanning
trivy image myapp:v1.0

# Scan for secrets
trivy fs --security-checks secret .
```

## Container Registries

### Image Distribution
Like **apartment blueprint libraries**:

#### Docker Hub (Public Library)
```bash
# Tag image for Docker Hub
docker tag myapp:v1.0 username/myapp:v1.0

# Push to Docker Hub
docker push username/myapp:v1.0

# Pull from Docker Hub
docker pull username/myapp:v1.0
```

#### Private Registry (Company Library)
```bash
# Tag for private registry
docker tag myapp:v1.0 registry.company.com/myapp:v1.0

# Login to private registry
docker login registry.company.com

# Push to private registry
docker push registry.company.com/myapp:v1.0
```

#### Registry Security
```bash
# Use registry with authentication
docker login -u username -p password registry.company.com

# Use image with digest (immutable reference)
```


## Advanced Container Concepts

### Init Systems
```dockerfile
# Use init system (apartment building manager)
FROM node:18-alpine

# Install tini (lightweight init system)
RUN apk add --no-cache tini

# Use tini as entrypoint
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["npm", "start"]
```

### Signal Handling
```dockerfile
# Proper signal handling (graceful tenant eviction)
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

```bash
#!/bin/bash
# entrypoint.sh - Graceful shutdown handling

# Function to handle shutdown
shutdown() {
    echo "Shutting down gracefully..."
    # Stop application gracefully
    kill -TERM "$child" 2>/dev/null
    wait "$child"
    exit 0
}

# Set up signal handlers
trap shutdown SIGTERM SIGINT

# Start application in background
npm start &
child=$!

# Wait for application or signals
wait "$child"
```

### Multi-Architecture Images
```bash
# Build for multiple architectures (universal apartment designs)
docker buildx create --name multiarch --use
docker buildx build --platform linux/amd64,linux/arm64 -t myapp:v1.0 --push .
```

## Container Best Practices

### Image Optimization
```dockerfile
# Use minimal base images (efficient apartment design)

# Minimize layers (efficient construction)

    npm ci --only=production && \
    npm cache clean --force

# Use .dockerignore (don't pack unnecessary items)
# .dockerignore file:
node_modules
.git
.gitignore
README.md
Dockerfile
.dockerignore
```

### Security Hardening
```dockerfile
# Security-hardened container (secure apartment)
FROM node:18-alpine

# Update packages
RUN apk update && apk upgrade

# Create non-root user
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001 -G nodejs

# Set working directory
WORKDIR /app

# Copy and install dependencies
COPY package*.json ./
RUN npm ci --only=production && \
    npm cache clean --force

# Copy application
COPY --chown=nextjs:nodejs . .

# Switch to non-root user
USER nextjs

# Use non-root port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# Start application
CMD ["npm", "start"]
```

### Development vs Production
```dockerfile
# Multi-stage build for different environments
FROM node:18-alpine AS base
WORKDIR /app
COPY package*.json ./

# Development stage
FROM base AS development
RUN npm ci
COPY . .
CMD ["npm", "run", "dev"]

# Production stage
FROM base AS production
RUN npm ci --only=production
COPY . .
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001 -G nodejs
RUN chown -R nextjs:nodejs /app
USER nextjs
CMD ["npm", "start"]
```

## Container Troubleshooting

### Common Issues and Solutions

#### Container Won't Start
```bash
# Check container logs
docker logs myapp-container

# Check image exists
docker images | grep myapp

# Check port conflicts
netstat -tulpn | grep :3000

# Run interactively for debugging
docker run -it --rm myapp:v1.0 /bin/sh
```

#### Performance Issues
```bash
# Check resource usage
docker stats myapp-container

# Check container processes
docker exec myapp-container ps aux

# Check disk usage
docker exec myapp-container df -h

# Check network connectivity
docker exec myapp-container ping google.com
```

#### Storage Issues
```bash
# Check volume mounts
docker inspect myapp-container | grep -A 10 Mounts

# Check disk space
docker system df

# Clean up unused resources
docker system prune -a
```

## Preparing for Kubernetes

### Container Design for Orchestration
```dockerfile
# Kubernetes-ready container (orchestration-friendly apartment)
FROM node:18-alpine

# Use environment variables for configuration
ENV NODE_ENV=production
ENV PORT=3000

# Create non-root user
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001 -G nodejs

WORKDIR /app

# Install dependencies
COPY package*.json ./
RUN npm ci --only=production

# Copy application
COPY --chown=nextjs:nodejs . .

# Switch to non-root user
USER nextjs

# Expose port
EXPOSE 3000

# Health check endpoint
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# Graceful shutdown support
CMD ["npm", "start"]
```

### Configuration Externalization
```dockerfile
# Externalize configuration (apartment settings from building management)
# Don't hardcode values - use environment variables
ENV DATABASE_URL=""
ENV REDIS_URL=""
ENV LOG_LEVEL="info"

# Use config files mounted as volumes
VOLUME ["/app/config"]

# Support configuration injection
CMD ["sh", "-c", "envsubst < /app/config/template.json > /app/config/config.json && npm start"]
```
