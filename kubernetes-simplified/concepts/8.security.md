# Kubernetes Security 🔐

## What is Kubernetes Security? (Simple Explanation)

Think of Kubernetes security like a **smart office building security system**!

### Real-Life Example 🏢
In a secure office building, you have:
- **ID Cards (Service Accounts)** - Employee identification
- **Access Levels (RBAC)** - Who can access what floors/rooms
- **Security Guards (Admission Controllers)** - Check permissions at doors
- **Building Rules (Pod Security Standards)** - Safety regulations for all offices
- **Visitor Badges (Tokens)** - Temporary access for guests
- **Security Cameras (Audit Logs)** - Track who did what and when

**Kubernetes security works the same way!**

## 1. Service Accounts 🆔 (Employee ID Cards)

### What are Service Accounts?
**Identity for pods** - like employee ID cards that identify who you are

**Real-Life Example:** Employee ID cards
- Every employee gets an ID card (every pod gets a service account)
- Different departments have different cards (different service accounts)
- Cards have access levels (permissions)
- Security guards check cards at doors (API server checks tokens)

### Default Service Account
```bash
# Every namespace gets a default service account
kubectl get serviceaccounts

# Expected output:
NAME      SECRETS   AGE
default   1         5d

# Check default service account details
kubectl describe serviceaccount default
```

### Custom Service Account Example
```yaml
# pizza-service-account.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: pizza-app-sa
  namespace: default
automountServiceAccountToken: true  # Automatically mount token in pods
---
# Secret for the service account (Kubernetes < 1.24)
apiVersion: v1
kind: Secret
metadata:
  name: pizza-app-sa-token
  annotations:
    kubernetes.io/service-account.name: pizza-app-sa
type: kubernetes.io/service-account-token
```

### Using Service Account in Pod
```yaml
# pod-with-sa.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-worker
spec:
  serviceAccountName: pizza-app-sa  # Use custom service account
  containers:
    - name: worker
      image: busybox:latest
      command: ['sleep', '3600']
```

### Step-by-Step: Create Employee ID Card
```bash
# Step 1: Create custom service account
kubectl apply -f pizza-service-account.yaml

# Step 2: Check service account
kubectl get serviceaccounts

# Step 3: Create pod with custom service account
kubectl apply -f pod-with-sa.yaml

# Step 4: Check mounted token in pod
kubectl exec pizza-worker -- ls -la /var/run/secrets/kubernetes.io/serviceaccount/

# You'll see:
# ca.crt (Certificate Authority)
# namespace (Current namespace)
# token (Authentication token)

# Step 5: Use token to access Kubernetes API
kubectl exec pizza-worker -- cat /var/run/secrets/kubernetes.io/serviceaccount/token
```

## 2. RBAC - Role-Based Access Control 🛡️ (Access Levels)

### What is RBAC?
**Permission system** - like access levels in an office building

**Real-Life Example:** Office access levels
- **Intern** - Can only access break room and their desk
- **Developer** - Can access development floor and meeting rooms
- **Manager** - Can access all floors and admin areas
- **Security** - Can access everywhere including security office

### RBAC Components
```
🛡️ RBAC PERMISSION SYSTEM

📋 ROLE (Job Description)
├── Lists what actions are allowed
├── "Can read files in accounting"
└── "Can create meetings in conference room"

👤 SUBJECT (Employee)
├── User (Human employee)
├── Group (Department)
└── ServiceAccount (Application)

🔗 ROLEBINDING (Assignment Letter)
├── Connects Role to Subject
├── "John Smith is assigned Developer role"
└── "Accounting app gets Accountant role"

🌐 CLUSTERROLE (Company-wide Position)
├── Permissions across all departments
└── "Security Chief can access all floors"

🔗 CLUSTERROLEBINDING (Company Assignment)
├── Assigns ClusterRole to Subject
└── "Jane Doe is Security Chief for entire company"
```

### Basic Role Example
```yaml
# pizza-roles.yaml
# Role for pizza developers (namespace-specific)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: pizza-dev
  name: pizza-developer
rules:
- apiGroups: [""]           # Core API group
  resources: ["pods"]       # Can work with pods
  verbs: ["get", "list", "create", "delete"]  # Allowed actions
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list"]    # Can only read services
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "create", "update", "patch"]
---
# RoleBinding - assign role to service account
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: pizza-developer-binding
  namespace: pizza-dev
subjects:
- kind: ServiceAccount
  name: pizza-app-sa        # Our service account
  namespace: pizza-dev
roleRef:
  kind: Role
  name: pizza-developer     # The role we created
  apiGroup: rbac.authorization.k8s.io
```

### Cluster-wide Role Example
```yaml
# pizza-cluster-roles.yaml
# ClusterRole for pizza administrators (cluster-wide)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: pizza-admin
rules:
- apiGroups: [""]
  resources: ["*"]          # All resources
  verbs: ["*"]              # All actions
- apiGroups: ["apps"]
  resources: ["*"]
  verbs: ["*"]
- apiGroups: ["networking.k8s.io"]
  resources: ["*"]
  verbs: ["*"]
---
# ClusterRoleBinding - assign cluster role
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: pizza-admin-binding
subjects:
- kind: User
  name: admin@pizza.com     # Human user
  apiGroup: rbac.authorization.k8s.io
- kind: ServiceAccount
  name: pizza-admin-sa
  namespace: kube-system
roleRef:
  kind: ClusterRole
  name: pizza-admin
  apiGroup: rbac.authorization.k8s.io
```

### Advanced RBAC Patterns

#### Aggregated ClusterRoles
```yaml
# aggregated-roles.yaml
# Base role for reading
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: base-reader
  labels:
    rbac.pizza.com/aggregate-to-reader: "true"
rules:
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list"]
---
# Extended role for apps
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: app-reader
  labels:
    rbac.pizza.com/aggregate-to-reader: "true"
rules:
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list"]
---
# Aggregated role (combines above)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: pizza-reader
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      rbac.pizza.com/aggregate-to-reader: "true"
rules: []  # Rules are automatically aggregated
```

#### Fine-grained Resource Access
```yaml
# fine-grained-rbac.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: pizza-operator
rules:
# Allow full access to pizza-related resources
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["*"]
  resourceNames: ["pizza-*"]  # Only pods starting with "pizza-"
# Allow reading secrets but only specific ones
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get"]
  resourceNames: ["pizza-db-secret", "pizza-api-secret"]
# Allow creating services but with restrictions
- apiGroups: [""]
  resources: ["services"]
  verbs: ["create", "update"]
# Allow patch operations on deployments
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "patch"]
  resourceNames: ["pizza-web", "pizza-api"]
```

#### Subject Access Review
```yaml
# subject-access-review.yaml
apiVersion: authorization.k8s.io/v1
kind: SubjectAccessReview
spec:
  resourceAttributes:
    namespace: production
    verb: delete
    group: apps
    version: v1
    resource: deployments
    name: pizza-web
  user: john@pizza.com
```

```bash
# Check access programmatically
kubectl auth can-i delete deployments/pizza-web --namespace=production --as=john@pizza.com

# Check what a user can do
kubectl auth can-i --list --as=john@pizza.com --namespace=production
```

## 3. Pod Security Standards 🏠 (Office Safety Rules)

### What are Pod Security Standards?
**Safety regulations** - like building safety codes that all offices must follow

**Real-Life Example:** Office safety rules
- **Privileged** - No safety rules (dangerous, like construction site)
- **Baseline** - Basic safety (fire exits, first aid kit)
- **Restricted** - Strict safety (security cameras, access controls, safety protocols)

### Pod Security Levels
```
🏠 OFFICE SAFETY LEVELS

🚧 PRIVILEGED (Construction Site)
├── No safety restrictions
├── Can do anything dangerous
├── Root access allowed
└── ⚠️ Use only for system components

🏢 BASELINE (Standard Office)
├── Basic safety measures
├── No root containers
├── No privileged escalation
└── ✅ Good for most applications

🏦 RESTRICTED (High Security Office)
├── Strict security measures
├── Non-root user required
├── Read-only root filesystem
├── No dangerous capabilities
└── 🔒 Best for production
```

### Pod Security Standard Examples

#### Privileged Pod (Dangerous)
```yaml
# privileged-pod.yaml - DON'T USE IN PRODUCTION!
apiVersion: v1
kind: Pod
metadata:
  name: dangerous-pod
spec:
  containers:
  - name: dangerous-container
    image: busybox:latest
    command: ['sleep', '3600']
    securityContext:
      privileged: true        # Can do anything!
      runAsUser: 0           # Runs as root
      allowPrivilegeEscalation: true
```

#### Baseline Pod (Standard Safety)
```yaml
# baseline-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: baseline-pod
spec:
  containers:
  - name: safe-container
    image: busybox:latest
    command: ['sleep', '3600']
    securityContext:
      runAsNonRoot: true      # Don't run as root
      runAsUser: 1000         # Run as specific user
      allowPrivilegeEscalation: false
      capabilities:
        drop:
        - ALL                 # Drop all dangerous capabilities
```

#### Restricted Pod (Maximum Security)
```yaml
# restricted-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: restricted-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault    # Use secure computing profile
  containers:
  - name: secure-container
    image: busybox:latest
    command: ['sleep', '3600']
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true  # Can't write to filesystem
      capabilities:
        drop:
        - ALL
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"
```

## 5. Secrets Security 🔐 (Safe Management)

### Secure Secret Handling
```yaml
# secure-secrets.yaml
apiVersion: v1
kind: Secret
metadata:
  name: pizza-secure-secrets
type: Opaque
data:
  database-password: <base64-encoded-password>
  api-key: <base64-encoded-key>
---
# Pod with secure secret usage
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  serviceAccountName: pizza-app-sa
  containers:
  - name: app
    image: nginx:latest
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: pizza-secure-secrets
          key: database-password
    volumeMounts:
    - name: secret-volume
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secret-volume
    secret:
      secretName: pizza-secure-secrets
      defaultMode: 0400     # Read-only for owner only
```

## 7. Security Best Practices 🌟

### 1. Principle of Least Privilege
- Give minimum permissions needed
- Use specific service accounts for different components
- Regularly audit and remove unused permissions

### 2. Pod Security
- Always use restricted pod security standards in production
- Run containers as non-root users
- Use read-only root filesystems when possible
- Drop all unnecessary capabilities

### 3. Network Security
- Implement default-deny network policies
- Allow only necessary traffic between components
- Use TLS for all communications

### 4. Secret Management
- Never store secrets in container images or ConfigMaps
- Use external secret management systems when possible
- Rotate secrets regularly
- Mount secrets as files with proper permissions

### 5. Monitoring and Auditing
- Enable audit logging
- Monitor for security events
- Set up alerts for suspicious activities
- Regular security scans of container images
