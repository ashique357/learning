# Kubernetes Configuration 📝

## What is Kubernetes Configuration? (Simple Explanation)

Think of Kubernetes configuration like **different ways to organize information in an office**!

### Real-Life Example 🏢
In an office, you have different ways to store information:
- **Recipe Book (ConfigMaps)** - Instructions and settings everyone can see
- **Safe (Secrets)** - Passwords and sensitive documents locked away
- **Environment Variables** - Quick notes on your desk
- **Command Line Arguments** - Specific instructions when starting a task
- **Resource Limits** - Budget constraints for each department

**Kubernetes configuration works the same way!**

## 1. ConfigMaps 📖 (Recipe Books)

### What are ConfigMaps?
**Non-sensitive configuration data** - like recipe books that everyone can read

**Real-Life Example:** Restaurant recipe book
- Contains cooking instructions (app settings)
- Everyone in kitchen can read it (not secret)
- Easy to update recipes (change configuration)
- Different books for different cuisines (different apps)

### Basic ConfigMap Example
```yaml
# app-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: pizza-config
data:
  # Simple key-value pairs
  app.mode: "production"
  debug.enabled: "false"
  max.connections: "100"

  # Multi-line configuration file
  database.properties: |
    host=db.pizza.com
    port=5432
    database=orders
    timeout=30

  # JSON configuration
  features.json: |
    {
      "online_ordering": true,
      "delivery_tracking": true,
      "loyalty_program": false
    }
```

### Step-by-Step: Create Recipe Book
```bash
# Step 1: Create ConfigMap from YAML
kubectl apply -f app-config.yaml

# Step 2: Check ConfigMap
kubectl get configmaps

# Expected output:
NAME           DATA   AGE
pizza-config   3      1m

# Step 3: See ConfigMap contents
kubectl describe configmap pizza-config

# Step 4: Create ConfigMap from command line
kubectl create configmap web-config \
  --from-literal=theme=dark \
  --from-literal=language=english \
  --from-file=nginx.conf

# Step 5: Create ConfigMap from directory
mkdir config-files
echo "server_name pizza.com;" > config-files/server.conf
echo "worker_processes 4;" > config-files/worker.conf
kubectl create configmap nginx-config --from-file=config-files/
```

### Using ConfigMap in Pods

#### Method 1: Environment Variables
```yaml
# pod-with-env.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-app
spec:
  containers:
  - name: app
    image: nginx:latest
    env:
    - name: APP_MODE              # Environment variable name
      valueFrom:
        configMapKeyRef:
          name: pizza-config      # ConfigMap name
          key: app.mode           # Key in ConfigMap
    - name: DEBUG_ENABLED
      valueFrom:
        configMapKeyRef:
          name: pizza-config
          key: debug.enabled
    # Load all keys as environment variables
    envFrom:
    - configMapRef:
        name: pizza-config
```

#### Method 2: Volume Mount (File System)
```yaml
# pod-with-volume.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pizza-app-files
spec:
  containers:
  - name: app
    image: nginx:latest
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config     # Mount path in container
    - name: specific-config
      mountPath: /etc/database.properties
      subPath: database.properties  # Mount specific file
  volumes:
  - name: config-volume
    configMap:
      name: pizza-config
  - name: specific-config
    configMap:
      name: pizza-config
```

### Step-by-Step: Use Recipe Book
```bash
# Step 1: Create pod with environment variables
kubectl apply -f pod-with-env.yaml

# Step 2: Check environment variables in pod
kubectl exec pizza-app -- env | grep -E "(APP_MODE|DEBUG_ENABLED)"

# Step 3: Create pod with file mounts
kubectl apply -f pod-with-volume.yaml

# Step 4: Check mounted files
kubectl exec pizza-app-files -- ls -la /etc/config

# Step 5: Read configuration file
kubectl exec pizza-app-files -- cat /etc/config/database.properties

# Step 6: Update ConfigMap (change recipe)
kubectl patch configmap pizza-config --patch '{"data":{"app.mode":"development"}}'

# Step 7: Check updated value (may need pod restart)
kubectl delete pod pizza-app
kubectl apply -f pod-with-env.yaml
kubectl exec pizza-app -- env | grep APP_MODE
```

## 2. Secrets 🔐 (Office Safe)

### What are Secrets?
**Sensitive configuration data** - like an office safe for passwords and important documents

**Real-Life Example:** Office safe
- Stores valuable items (passwords, certificates)
- Only authorized people have access
- Contents are protected/encrypted
- Different compartments for different secrets

### Secret Types
```
🔐 OFFICE SAFE COMPARTMENTS

📄 Opaque (Generic Documents)
├── Passwords
├── API keys
└── Custom sensitive data

🌐 TLS (Security Certificates)
├── SSL certificates
├── Private keys
└── Certificate chains

🐳 Docker Registry (ID Cards)
├── Docker username
├── Docker password
└── Registry access tokens

🔑 Service Account (Employee Badges)
├── Kubernetes API tokens
├── Service permissions
└── Automatic mounting

📱 Basic Auth (Login Credentials)
├── Username
└── Password

🔑 SSH Auth (SSH Keys)
├── Private SSH key
└── Known hosts
```

### Secret Encryption at Rest
**Advanced security features:**
- **etcd Encryption** - Secrets encrypted in etcd database
- **Envelope Encryption** - Multiple layers of encryption
- **Key Management Service (KMS)** - Cloud provider key management
- **Sealed Secrets** - GitOps-friendly encrypted secrets

```yaml
# kms-encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration

- resources:
  - secrets
  providers:
  - kms:
      name: aws-kms
      endpoint: unix:///tmp/socketfile.sock
      cachesize: 100
      timeout: 3s
  - identity: {}  # Fallback to unencrypted
```

### Basic Secret Example
```yaml
# app-secrets.yaml
apiVersion: v1
kind: Secret

  name: pizza-secrets
```

### Creating Secrets (Different Methods)

#### Method 1: From Command Line (Automatic Encoding)
```bash
# Create secret from literals (automatically base64 encoded)
kubectl create secret generic pizza-secrets \
  --from-literal=database-password=pizza-db-pass \
  --from-literal=api-key=abcdefghijk \
  --from-literal=admin-token=super-secret-token

# Create secret from files
echo -n 'pizza-db-pass' > db-password.txt
echo -n 'abcdefghijk' > api-key.txt
kubectl create secret generic pizza-file-secrets \
  --from-file=database-password=db-password.txt \
  --from-file=api-key=api-key.txt
```

#### Method 2: Manual Base64 Encoding
```bash
# Encode values manually
echo -n 'pizza-db-pass' | base64    # Output: cGl6emEtZGItcGFzcw==
echo -n 'abcdefghijk' | base64      # Output: YWJjZGVmZ2hpams=

# Then use in YAML file
```

#### Method 3: TLS Secret
```bash
# Create TLS secret from certificate files
kubectl create secret tls pizza-tls-secret \
  --cert=pizza.crt \
  --key=pizza.key
```

#### Method 4: Docker Registry Secret
```bash
# Create Docker registry secret
kubectl create secret docker-registry pizza-registry-secret \
  --docker-server=registry.pizza.com \
  --docker-username=pizzauser \
  --docker-password=pizzapass \
  --docker-email=admin@pizza.com
```

```yaml
# field-ref-examples.yaml

  name: field-ref-demo

          fieldPath: metadata.namespace
    - name: POD_IP
      valueFrom:

      valueFrom:
        fieldRef:
          fieldPath: spec.nodeName
    - name: SERVICE_ACCOUNT
      valueFrom:
        fieldRef:
          fieldPath: spec.serviceAccountName
    
    # Pod labels and annotations
    - name: APP_VERSION
      valueFrom:
        fieldRef:
          fieldPath: metadata.labels['version']
    
    # Resource information
    - name: CPU_REQUEST
      valueFrom:
        resourceFieldRef:

    - name: MEMORY_LIMIT
      valueFrom:
        resourceFieldRef:
```

### Advanced Resource Management

#### Quality of Service (QoS) Classes Deep Dive
```yaml
# qos-examples.yaml
# Guaranteed QoS (Highest Priority)
apiVersion: v1
kind: Pod
metadata:
  name: guaranteed-pod
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        memory: "1Gi"
        cpu: "500m"
      limits:
        memory: "1Gi"    # Same as requests
        cpu: "500m"      # Same as requests
---
# Burstable QoS (Medium Priority)
apiVersion: v1
kind: Pod
metadata:
  name: burstable-pod
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        memory: "512Mi"
        cpu: "250m"
      limits:
        memory: "1Gi"    # Higher than requests
        cpu: "1000m"     # Higher than requests
---
# BestEffort QoS (Lowest Priority)
apiVersion: v1
kind: Pod
metadata:
  name: besteffort-pod
spec:
  containers:
  - name: app
    image: nginx
    # No resources specified
```

```yaml
# vpa-example.yaml

      controlledResources: ["cpu", "memory"]
      controlledValues: RequestsAndLimits
```

### Step 2: Web Application Deployment
```yaml
# web-app-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pizza-web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: pizza-web
  template:
    metadata:
      labels:
        app: pizza-web
    spec:
      containers:
      - name: web-app
        image: nginx:latest
        ports:
        - containerPort: 8080
        
        # Environment variables from multiple sources
        env:
        - name: APP_NAME
          value: "Pizza Ordering System"
        - name: ENVIRONMENT
          value: "production"
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: web-app-secrets
              key: database-url
        - name: STRIPE_API_KEY
          valueFrom:
            secretKeyRef:
              name: web-app-secrets
              key: stripe-api-key
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        
        # Load all ConfigMap keys as environment variables
        envFrom:
        - configMapRef:
            name: web-app-config
            prefix: CONFIG_
        
        # Mount configuration files
        volumeMounts:
        - name: app-config
          mountPath: /etc/config
        - name: nginx-config
          mountPath: /etc/nginx/conf.d/default.conf
          subPath: nginx.conf
        - name: app-secrets
          mountPath: /etc/secrets
          readOnly: true
        
        # Resource management
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        
        # Health checks
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
        
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
      
      volumes:
      - name: app-config
        configMap:
          name: web-app-config
      - name: nginx-config
        configMap:
          name: web-app-config
      - name: app-secrets
        secret:
          secretName: web-app-secrets
          defaultMode: 0400
---
apiVersion: v1
kind: Service
metadata:
  name: pizza-web-service
spec:
  selector:
    app: pizza-web
  ports:
  - port: 80
    targetPort: 8080
  type: LoadBalancer
```

### Step 3: Deploy and Test
```bash

kubectl apply -f web-app-config.yaml

# Step 2: Deploy application
```

## 6. Configuration Best Practices 🌟

### 1. Separation of Concerns
```yaml
# Good: Separate configs by purpose
kind: ConfigMap
metadata:
  name: app-config        # Application settings
---
kind: ConfigMap
metadata:
  name: database-config   # Database settings
---
kind: Secret
metadata:
  name: app-secrets       # Sensitive data only
```

### 2. Use Appropriate Methods
- **ConfigMaps** - Non-sensitive configuration
- **Secrets** - Passwords, API keys, certificates
- **Environment Variables** - Simple key-value pairs
- **Volume Mounts** - Configuration files

### 3. Resource Management
```yaml
resources:
  requests:     # What you need
    memory: "256Mi"
    cpu: "250m"
  limits:       # Maximum allowed
    memory: "512Mi"
    cpu: "500m"
```

### 4. Security
- Never put secrets in ConfigMaps
- Use proper file permissions for mounted secrets
- Rotate secrets regularly
- Use service accounts for API access
