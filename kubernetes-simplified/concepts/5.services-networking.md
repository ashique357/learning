# Kubernetes Services and Networking 🌐

## What is Kubernetes Networking? (Simple Explanation)

Think of Kubernetes networking like a **smart city's communication system**!

### Real-Life Example 🏙️
In a city, you need different ways to communicate:
- **Phone System (Services)** - Stable phone numbers to reach businesses
- **Postal Service (Ingress)** - Routes mail from outside the city
- **Internal Roads (Pod Network)** - How people move between buildings
- **Traffic Rules (Network Policies)** - Control who can go where
- **Address Book (DNS)** - Find businesses by name, not just address

**Kubernetes networking works the same way!**

## 1. Pod Networking 🏠 (Building Addresses)

### How Pod Networking Works
Every pod gets its own IP address - like every building having a unique address!

**Real-Life Example:** Apartment buildings
- Each building has unique address (10.1.1.5, 10.1.1.6)
- People can visit any building directly
- All buildings are on the same street network

### Pod Network Example
```yaml
# Create two pods to test networking
# pod1.yaml
apiVersion: v1
kind: Pod
metadata:
  name: building-1
  labels:
    app: web
spec:
  containers:
    - name: web-server
      image: nginx:latest
      ports:
        - containerPort: 80
---
# pod2.yaml  
apiVersion: v1
kind: Pod
metadata:
  name: building-2
  labels:
    app: client
spec:
  containers:
    - name: client
      image: busybox:latest
      command: ['sleep', '3600']
```

### Step-by-Step: Test Pod Communication
```bash
# Step 1: Create both pods
kubectl apply -f pod1.yaml
kubectl apply -f pod2.yaml

# Step 2: Get pod IPs
kubectl get pods -o wide

# Expected output:
NAME         READY   STATUS    RESTARTS   AGE   IP           NODE
building-1   1/1     Running   0          1m    10.244.1.5   worker-1
building-2   1/1     Running   0          1m    10.244.2.3   worker-2

# Step 3: Test communication from building-2 to building-1
kubectl exec building-2 -- wget -qO- http://10.244.1.5

# You should see nginx welcome page!
```

### Pod Network Rules 📋
1. **Every pod gets unique IP** (like unique building address)
2. **Pods can talk to any other pod** (no firewalls by default)
3. **Pod IP changes when pod restarts** (like moving to new building)
4. **Same node or different node - doesn't matter** (city-wide network)

## 2. Services 📞 (Phone System)

### What is a Service?
**Stable way to reach your apps** - like a business phone number that never changes

**Real-Life Example:** Pizza restaurant
- Restaurant might move buildings (pod restarts)
- But phone number stays the same (service)
- Customers always call same number
- Phone system routes to current location

### Service Types Overview
```
📞 SERVICE TYPES (Phone Systems)

🏠 ClusterIP (Internal Phone)
├── Only works inside the city
├── Other businesses can call
└── Customers outside city cannot call

🚪 NodePort (Building Phone)  
├── Phone installed on building entrance
├── Outside customers can call
└── Uses building's external number + extension

☁️ LoadBalancer (Reception Desk)
├── Professional reception service
├── One number for outside customers
└── Routes calls to available restaurants

🌐 ExternalName (Call Forwarding)
├── Forwards calls to external business
├── Like "Press 1 for Pizza Hut"
└── Points to business outside the city
```

### ClusterIP Service (Internal Phone) 🏠

**Most common service type** - for internal communication

```yaml
# pizza-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: pizza-service
spec:
  type: ClusterIP          # Internal only (default)
  selector:
    app: pizza             # Connect to pods with label app=pizza
  ports:
  - port: 80              # Service listens on port 80
    targetPort: 8080      # Forward to pod port 8080
    protocol: TCP
```

### Step-by-Step: Create Internal Phone System
```bash
# Step 1: Create pizza deployment
kubectl create deployment pizza-shop --image=nginx --replicas=3

# Step 2: Label the pods
kubectl label deployment pizza-shop app=pizza

# Step 3: Create service
kubectl apply -f pizza-service.yaml

# Step 4: Check service
kubectl get services

# Expected output:
NAME            TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
pizza-service   ClusterIP   10.96.123.45    <none>        80/TCP    1m

# Step 5: Test from inside cluster
kubectl run test-client --image=busybox --rm -it -- sh
# Inside the pod:
wget -qO- http://pizza-service
# You'll see nginx page!
```

### How Service Routes Traffic
```
📞 SERVICE ROUTING MAGIC

Customer calls: pizza-service:80
                     │
            ┌────────┼────────┐
            │   SERVICE        │
            │  (Phone System)  │
            └────────┼────────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ Pod 1   │ │ Pod 2   │ │ Pod 3   │
   │ :8080   │ │ :8080   │ │ :8080   │
   └─────────┘ └─────────┘ └─────────┘
   
Service automatically balances calls between all pods!
```

### NodePort Service (Building Phone) 🚪

**Access from outside cluster** - like phone at building entrance

```yaml
# pizza-nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: pizza-nodeport
spec:
  type: NodePort
  selector:
    app: pizza
  ports:
  - port: 80              # Service port
    targetPort: 8080      # Pod port  
    nodePort: 30080       # External port (30000-32767)
```

### Step-by-Step: Install Building Phone
```bash
# Step 1: Create NodePort service
kubectl apply -f pizza-nodeport.yaml

# Step 2: Check service
kubectl get services

# Expected output:
NAME             TYPE       CLUSTER-IP     EXTERNAL-IP   PORT(S)        AGE
pizza-nodeport   NodePort   10.96.45.67    <none>        80:30080/TCP   1m

# Step 3: Get node IP
kubectl get nodes -o wide

# Step 4: Access from outside (replace with your node IP)
curl http://192.168.1.100:30080
# You can access from your browser too!
```

### LoadBalancer Service (Reception Desk) ☁️

**Professional external access** - cloud provider creates load balancer

```yaml
# pizza-loadbalancer.yaml
apiVersion: v1
kind: Service
metadata:
  name: pizza-loadbalancer
spec:
  type: LoadBalancer
  selector:
    app: pizza
  ports:
  - port: 80
    targetPort: 8080
```

### Step-by-Step: Hire Reception Service
```bash

kubectl apply -f pizza-loadbalancer.yaml

# Step 2: Wait for external IP (takes 1-2 minutes)

pizza-loadbalancer    LoadBalancer   10.96.78.90    34.123.45.67    80:31234/TCP   2m

# Step 3: Access using external IP
curl http://34.123.45.67
```

## 3. Ingress 📮 (Post Office)

### What is Ingress?
**Smart routing from internet** - like a post office that sorts mail by address

**Real-Life Example:** City post office
- All mail comes to one address (load balancer)
- Post office reads addresses and routes mail
- Different streets (paths) go to different neighborhoods
- Different cities (domains) handled by same office

### Ingress Architecture
```
📮 INGRESS ROUTING SYSTEM

🌍 Internet Traffic
        │
   ┌─────────┐
   │ 📮 INGRESS │ ← One entry point
   │ Controller │
   └─────────┘
        │
   ┌────┴────┐
   │ ROUTING │
   │ RULES   │
   └────┬────┘
        │
   ┌────┴─────────────────┐
   │                      │
┌─────────┐        ┌─────────┐
│ 📞 Web  │        │ 📞 API  │
│ Service │        │ Service │
└─────────┘        └─────────┘
   │                      │
┌─────────┐        ┌─────────┐
│ 🏠 Web  │        │ 🏠 API  │
│ Pods    │        │ Pods    │
└─────────┘        └─────────┘
```

### Basic Ingress Example
```yaml
# pizza-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: pizza-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: pizza.example.com        # Domain name
    http:
      paths:
      - path: /                    # All paths
        pathType: Prefix
        backend:
          service:
            name: pizza-service    # Route to this service
            port:
              number: 80
  - host: api.pizza.example.com    # Different domain
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: pizza-api-service
            port:
              number: 80
```

```yaml

kind: IngressClass
```

#### Multiple Ingress Controllers
```bash

# Nginx for external traffic

  --set controller.ingressClass=nginx-external \
  --set controller.service.type=LoadBalancer

# Traefik for internal traffic

  --set ingressClass.name=traefik-internal \
  --set service.type=ClusterIP

# HAProxy for API traffic
helm install haproxy-api haproxytech/kubernetes-ingress \
  --set controller.ingressClass=haproxy-api
```

## 4. DNS 📋 (Address Book)

### How Kubernetes DNS Works
**Automatic name resolution** - like a smart address book

**Real-Life Example:** Phone book
- Instead of remembering phone numbers, you use names
- "Call Pizza Palace" instead of "Call 555-1234"
- Phone book automatically looks up the number

### DNS Implementation Details
**CoreDNS** - The DNS server in Kubernetes
- **Cluster DNS** - Runs as pods in kube-system namespace
- **DNS Policy** - Controls how pods resolve DNS
- **DNS Config** - Custom DNS settings for pods
- **Service Discovery** - Automatic service registration

### DNS Naming Rules
```
📋 KUBERNETES DNS ADDRESS BOOK

Service Name: pizza-service
Namespace: default
Full DNS Name: pizza-service.default.svc.cluster.local

┌─────────────────┬──────────────┬─────────────┬──────────────┐
│   Service       │  Namespace   │   Type      │   Domain     │
│   Name          │              │             │              │
├─────────────────┼──────────────┼─────────────┼──────────────┤
│ pizza-service   │   default    │    svc      │cluster.local │
│ web-service     │   frontend   │    svc      │cluster.local │
│ db-service      │   backend    │    svc      │cluster.local │
└─────────────────┴──────────────┴─────────────┴──────────────┘
```

### DNS Record Types
```
📋 DNS RECORD TYPES

🔍 A Records (IPv4)
├── pizza-service.default.svc.cluster.local → 10.96.123.45
├── web-pod.default.pod.cluster.local → 10.244.1.5
└── node1.cluster.local → 192.168.1.100

🔍 AAAA Records (IPv6)
├── pizza-service.default.svc.cluster.local → 2001:db8::1
└── (when IPv6 is enabled)

🔍 SRV Records (Service Discovery)
├── _http._tcp.pizza-service.default.svc.cluster.local
├── Priority: 0, Weight: 100, Port: 80
└── Target: pizza-service.default.svc.cluster.local

🔍 PTR Records (Reverse DNS)
├── 45.123.96.10.in-addr.arpa → pizza-service.default.svc.cluster.local
└── (for reverse lookups)
```

### DNS Policies
```yaml
# Different DNS policies for pods
apiVersion: v1
kind: Pod
metadata:
  name: dns-policy-examples
spec:
  dnsPolicy: ClusterFirst    # Default - use cluster DNS first
  # Other options:
  # Default - use node's DNS
  # ClusterFirstWithHostNet - cluster DNS + host network
  # None - custom DNS config only
  
  dnsConfig:                 # Custom DNS configuration
    nameservers:
    - 8.8.8.8
    - 8.8.4.4
    searches:
    - my-company.com
    - example.com
    options:
    - name: ndots
      value: "2"
    - name: edns0
  containers:
  - name: app
    image: nginx
```


## 5. Network Policies 🚦 (Traffic Rules)

### What are Network Policies?
**Traffic control rules** - like traffic lights and road signs

**Real-Life Example:** City traffic rules
- Some roads are one-way only
- Trucks can't go on residential streets
- Emergency vehicles have special access
- School zones have restricted access

### Default Behavior (No Traffic Rules)
```
🚦 NO NETWORK POLICIES = NO TRAFFIC RULES

┌─────────┐    ┌─────────┐    ┌─────────┐
│ Web App │◄──►│   API   │◄──►│Database │
└─────────┘    └─────────┘    └─────────┘
     ▲              ▲              ▲
     │              │              │
┌─────────┐    ┌─────────┐    ┌─────────┐
│ Hacker  │    │ Random  │    │ Anyone  │
│ Pod     │    │ Pod     │    │ Can     │
└─────────┘    └─────────┘    └─────────┘

Everyone can talk to everyone! 😱
```

### Network Policy Implementation
**CNI Plugin Requirement:**
- **Calico** - Full network policy support
- **Cilium** - eBPF-based policies
- **Weave Net** - Basic policy support
- **Flannel** - No policy support (needs additional components)

### Network Policy Example (Database Protection)
```yaml
# protect-database.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: protect-database
spec:
  podSelector:
    matchLabels:
      app: database           # Apply to database pods
  policyTypes:
  - Ingress                   # Control incoming traffic
  - Egress                    # Control outgoing traffic
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api            # Only allow API pods
    - namespaceSelector:
        matchLabels:
          name: backend       # Only from backend namespace
    ports:
    - protocol: TCP
      port: 3306              # Only on database port
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          name: kube-system   # Allow DNS queries
    ports:
    - protocol: UDP
      port: 53
```

### Advanced Network Policy Patterns

#### Micro-segmentation
```yaml
# micro-segmentation.yaml
# Deny all traffic by default
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}             # Apply to all pods
  policyTypes:
  - Ingress
  - Egress
  # No ingress/egress rules = deny all
---
# Allow specific communication patterns
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: web
    ports:
    - protocol: TCP
      port: 8080
```

#### CIDR-based Policies
```yaml
# cidr-policy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-access
spec:
  podSelector:
    matchLabels:
      app: public-api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - ipBlock:
        cidr: 10.0.0.0/8        # Allow internal network
        except:
        - 10.1.1.0/24           # Except this subnet
    - ipBlock:
        cidr: 192.168.1.0/24    # Allow office network
    ports:
    - protocol: TCP
      port: 443
```

#### Time-based Policies (with Calico)
```yaml
# time-based-policy.yaml (Calico CRD)
apiVersion: projectcalico.org/v3
kind: NetworkPolicy
metadata:
  name: business-hours-only
  namespace: production
spec:
  selector: app == "business-app"
  types:
  - Ingress
  ingress:
  - action: Allow
    source:
      selector: role == "client"
    destination:
      ports:
      - 8080
    # Only during business hours (9 AM - 5 PM UTC)
    schedule: "0 9 * * 1-5 - 0 17 * * 1-5"
```

### Common Network Policy Patterns

#### 1. Deny All Traffic (Lockdown)
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}             # Apply to all pods
  policyTypes:
  - Ingress
  - Egress
  # No ingress/egress rules = deny all
```

#### 2. Allow Only Same Namespace
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-same-namespace
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: production    # Only from production namespace
```

```yaml
            matchLabels:
              name: pizza-frontend

            matchLabels:
              app: database

        - protocol: TCP
```

## Networking Best Practices 🌟

### 1. Service Design
- Use ClusterIP for internal communication
- Use LoadBalancer for external access in cloud
- Use meaningful service names (pizza-api, not service-1)

### 2. Security
- Always use Network Policies in production
- Start with deny-all, then allow specific traffic
- Separate namespaces for different environments

### 3. DNS
- Use service names instead of IP addresses
- Include namespace for cross-namespace communication
- Keep service names short and descriptive

### 4. Ingress
- Use one ingress controller per cluster
- Group related services under same domain
- Use TLS certificates for production
