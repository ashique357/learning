# Kubernetes Cluster Architecture 🏗️

## What is a Kubernetes Cluster? (Simple Explanation)

Think of a Kubernetes cluster like a **smart city with a government**!

### Real-Life Example 🏙️
Imagine a city:
- **City Hall (Control Plane)** - Makes all the decisions
- **Neighborhoods (Worker Nodes)** - Where people actually live
- **Mayor (API Server)** - Takes all requests and commands
- **City Records (etcd)** - Keeps track of everything
- **City Planner (Scheduler)** - Decides where new buildings go
- **Inspectors (Controllers)** - Make sure everything works as planned

## Control Plane Components (The Government) 🏛️

### 1. API Server 📞 (The Mayor)

**What it does:** Takes all your requests and commands

**Real-Life Example:** Like a mayor who:
- Listens to citizen requests
- Checks if requests are valid
- Passes approved requests to the right department

```bash
# When you run this command:
kubectl get pods

# Here's what happens:
You → API Server → "Show me all pods"
API Server → etcd → "Get pod information"
etcd → API Server → "Here's the data"
API Server → You → "Here are your pods"
```

**Step-by-Step Process:**
```
1. 👤 You: "kubectl get pods"
2. 📞 API Server: "Let me check if you're allowed"
3. 📞 API Server: "Authentication ✅, Authorization ✅"
4. 📞 API Server: "etcd, give me pod data"
5. 📋 etcd: "Here's all pod information"
6. 📞 API Server: "Here's your result!"
```

### 2. etcd 📋 (City Records)

**What it does:** Stores ALL information about your cluster

**Real-Life Example:** Like city records that store:
- Who lives where (which pods are on which nodes)
- Building permits (deployments, services)
- City rules (policies, configurations)

```bash
# etcd stores everything as key-value pairs
/registry/pods/default/my-app-123 → {pod details}
/registry/services/default/my-service → {service details}
/registry/nodes/worker-node-1 → {node details}
```

**What's Stored in etcd:**
```
📋 CLUSTER STATE DATABASE
├── 🏠 Pods (where they are, what they're doing)
├── 🏢 Nodes (available servers)
├── ⚙️ Services (how to reach apps)
├── 🔐 Secrets (passwords, certificates)
├── 📝 ConfigMaps (configuration files)
├── 👥 Users & Permissions
└── 📊 Everything else!
```

### 3. Scheduler 🎯 (City Planner)

**What it does:** Decides WHERE to place new pods

**Real-Life Example:** Like a city planner who:
- Looks at available land (nodes)
- Checks zoning rules (node selectors, taints)
- Finds the best location for new buildings (pods)

```yaml
# When you create a pod:
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: nginx
      resources:
        requests:
          memory: "64Mi"
          cpu: "250m"
```

**Scheduler's Decision Process:**
```
🎯 SCHEDULER THINKING:
1. "New pod needs: 64Mi memory, 250m CPU"
2. "Let me check all nodes..."
   
   Node 1: 2GB free memory, 1 CPU free ✅
   Node 2: 32Mi free memory, 0.1 CPU free ❌ (not enough)
   Node 3: 4GB free memory, 2 CPU free ✅
   
3. "Node 1 and Node 3 can fit it"
4. "Node 3 has more resources, I'll choose that"
5. "Assigning pod to Node 3!"
```

### 4. Controller Manager 🔍 (City Inspectors)

**What it does:** Makes sure everything works as you want it

**Real-Life Example:** Like city inspectors who:
- Check if buildings meet safety codes
- Fix problems automatically
- Ensure city rules are followed

```yaml
# You want 3 copies of your app:
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3  # I want 3 pods
```

**Controller's Monitoring Loop:**
```
🔍 CONTROLLER THINKING (every few seconds):
1. "User wants 3 pods running"
2. "Let me count current pods..."
3. "I see only 2 pods running"
4. "I need to create 1 more pod"
5. "Creating new pod..."
6. "Now I have 3 pods ✅"

# If a pod crashes:
1. "Oh no! Only 2 pods running now"
2. "I need to create 1 replacement"
3. "Creating new pod..."
4. "Back to 3 pods ✅"
```

**Built-in Controllers:**
- **Deployment Controller** - Manages ReplicaSets and rolling updates
- **ReplicaSet Controller** - Ensures desired number of pod replicas
- **StatefulSet Controller** - Manages stateful applications
- **DaemonSet Controller** - Ensures pods run on all/selected nodes
- **Job Controller** - Manages run-to-completion tasks
- **CronJob Controller** - Manages time-based jobs
- **Service Controller** - Manages service endpoints
- **Node Controller** - Manages node lifecycle
- **Namespace Controller** - Handles namespace lifecycle
- **PersistentVolume Controller** - Manages storage lifecycle

**Custom Controllers & Operators:**
- **Custom Resource Definitions (CRDs)** - Define new API objects
- **Operators** - Domain-specific controllers (databases, monitoring)
- **Controller Runtime** - Framework for building controllers
- **Operator SDK** - Tools for operator development

## Worker Node Components (The Neighborhoods) 🏢

### 1. kubelet 🤖 (Building Manager)

**What it does:** Manages all pods on this node

**Real-Life Example:** Like a building manager who:
- Gets instructions from city hall
- Makes sure apartments are ready
- Reports back on building status

```bash
# kubelet's daily routine:
1. "Let me check with API Server for new instructions"
2. "API Server says: Run nginx pod with 1GB memory"
3. "I'll tell Container Runtime to start it"
4. "Pod is running, let me report back to API Server"
5. "Checking pod health... still running ✅"
```

**kubelet Communication:**
```
🤖 KUBELET ↔️ 📞 API SERVER
│
├── "What pods should I run?"
├── "Pod xyz is running healthy"
├── "Pod abc crashed, restarting..."
└── "Node has 4GB memory available"
```

### 2. kube-proxy 🌐 (Network Manager)

**What it does:** Handles network traffic to your pods

**Real-Life Example:** Like a postal service that:
- Knows all addresses in the neighborhood
- Routes mail to the right apartment
- Updates addresses when people move

```yaml
# When you create a service:
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
  ports:
  - port: 80
    targetPort: 8080
```

**kube-proxy's Job:**
```
🌐 KUBE-PROXY ROUTING:
1. "Service my-app-service should go to pods with label app=my-app"
2. "I found 3 pods: 10.1.1.5, 10.1.2.3, 10.1.3.7"
3. "Setting up routing rules..."
4. "Traffic to my-app-service will be distributed to these 3 pods"

When traffic comes:
User Request → Service (my-app-service:80)
            → kube-proxy routes to → Pod 1 (10.1.1.5:8080)
                                  → Pod 2 (10.1.2.3:8080)  
                                  → Pod 3 (10.1.3.7:8080)
```

**kube-proxy Modes:**
- **iptables mode** (default)
    - Uses Linux iptables rules
    - Good performance for most clusters
    - Random load balancing
- **IPVS mode** (advanced)
    - Uses Linux IP Virtual Server
    - Better performance for large clusters
    - Multiple load balancing algorithms
    - Requires IPVS kernel modules
- **userspace mode** (legacy)
    - Proxy runs in userspace
    - Slower but more compatible
    - Rarely used in modern clusters

**Load Balancing Algorithms (IPVS):**
- **Round Robin** - Distribute requests evenly
- **Least Connection** - Send to pod with fewest connections
- **Source Hashing** - Consistent routing based on client IP
- **Weighted Round Robin** - Distribute based on pod capacity

**Service Types Handled:**
- **ClusterIP** - Internal cluster communication
- **NodePort** - External access via node ports
- **LoadBalancer** - Cloud provider load balancers
- **ExternalName** - DNS CNAME records

### 3. Container Runtime 📦 (Apartment Builder)

**What it does:** Actually runs your containers

**Real-Life Example:** Like construction workers who:
- Build apartments according to blueprints
- Provide utilities (CPU, memory, storage)
- Maintain the living spaces

**Container Runtime Interface (CRI):**
- **Standard interface** between kubelet and container runtime
- **Pluggable architecture** - swap runtimes without changing Kubernetes
- **gRPC-based API** for container and image management

**Popular Container Runtimes:**
- **containerd** 📦 (Most common, CNCF graduated)
    - Industry standard, lightweight
    - Used by Docker Desktop, cloud providers
    - Direct CRI implementation
- **CRI-O** 🔧 (Kubernetes-focused)
    - Built specifically for Kubernetes
    - OCI-compliant, minimal overhead
    - Red Hat/OpenShift default
- **Docker Engine** 🐳 (Legacy support)
    - Popular but deprecated in Kubernetes
    - Uses dockershim (being removed)
    - Still works via containerd

**Container Runtime Workflow:**
```bash
# Container Runtime workflow:
1. kubelet: "Please run nginx:latest container"
2. Container Runtime: "Downloading nginx:latest image..."
3. Container Runtime: "Creating container with 1GB memory limit"
4. Container Runtime: "Container is running with PID 1234"
5. kubelet: "Great! Container is healthy"
```

**Low-Level Runtimes:**
- **runc** - OCI reference implementation
- **crun** - Fast, low-memory C implementation
- **kata-containers** - Secure containers with VMs
- **gVisor** - User-space kernel for containers

**Container Lifecycle:**
```
Image Pull → Container Create → Container Start → Container Running → Container Stop → Container Remove
```

## How Everything Works Together 🔄

### Example: Deploying a New Application

**Step 1: You Give Command**
```bash
kubectl create deployment my-web-app --image=nginx --replicas=2
```

**Step 2: The Journey Begins**
```
👤 You → 📞 API Server: "Create deployment with 2 nginx pods"
📞 API Server → 📋 etcd: "Save this deployment info"
📞 API Server → 🔍 Controller: "New deployment created"
🔍 Controller → 📞 API Server: "I need to create 2 pods"
📞 API Server → 🎯 Scheduler: "Find homes for 2 pods"
🎯 Scheduler → 📞 API Server: "Put pod1 on node1, pod2 on node2"
📞 API Server → 🤖 kubelet (node1): "Run nginx pod"
📞 API Server → 🤖 kubelet (node2): "Run nginx pod"
🤖 kubelet → 📦 Container Runtime: "Start nginx container"
📦 Container Runtime: "Container running!"
🤖 kubelet → 📞 API Server: "Pod is healthy and running"
```

**Visual Flow:**
```
1. 👤 kubectl command
   ↓
2. 📞 API Server (validates & stores)
   ↓
3. 🔍 Controller (creates pods)
   ↓
4. 🎯 Scheduler (assigns nodes)
   ↓
5. 🤖 kubelet (starts containers)
   ↓
6. 📦 Container Runtime (runs app)
   ↓
7. ✅ Your app is running!
```

## High Availability (Multiple Control Planes) 🏛️🏛️

For production, you want multiple control planes:

```
🏙️ HIGHLY AVAILABLE CLUSTER
┌─────────────────────────────────────────────────────────────┐
│                 🏛️ CONTROL PLANE 1                         │
│ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌────────┐ │
│ │📞 API Server│ │📋 etcd      │ │🎯 Scheduler │ │🔍 Ctrl │ │
│ └─────────────┘ └─────────────┘ └─────────────┘ └────────┘ │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                 🏛️ CONTROL PLANE 2                         │
│ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌────────┐ │
│ │📞 API Server│ │📋 etcd      │ │🎯 Scheduler │ │🔍 Ctrl │ │
│ └─────────────┘ └─────────────┘ └─────────────┘ └────────┘ │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                 🏛️ CONTROL PLANE 3                         │
│ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌────────┐ │
│ │📞 API Server│ │📋 etcd      │ │🎯 Scheduler │ │🔍 Ctrl │ │
│ └─────────────┘ └─────────────┘ └─────────────┘ └────────┘ │
└─────────────────────────────────────────────────────────────┘
                              │
                    ┌─────────┼─────────┐
                    │         │         │
            🏢 WORKER NODES (same as before)
```

**Why 3 Control Planes?**
- If 1 fails, you still have 2 working ✅
- etcd needs odd numbers for voting (3, 5, 7)
- Your cluster keeps running even during maintenance

## Common Architecture Patterns 🏗️

### 1. Development Cluster (Simple)
```
1 Control Plane + 1 Worker Node
Perfect for learning and testing
```

### 2. Production Cluster (Recommended)
```
3 Control Planes + 3+ Worker Nodes
High availability and scalability
```

### 3. Edge Cluster (Lightweight)
```
1 Control Plane + 2 Worker Nodes
For remote locations or IoT
```

## Troubleshooting Architecture Issues 🔧

### Problem: Node Not Ready
```bash
kubectl describe node worker-node-1
```
**Common causes:**
- kubelet not running
- Network issues
- Resource exhaustion

### Problem: Control Plane Issues
```bash
kubectl get componentstatuses
```
**Common causes:**
- etcd connectivity problems
- Certificate issues
- Resource constraints

### Problem: Pods Stuck in Pending
```bash
kubectl describe pod my-pod
```
**Common causes:**
- No available nodes
- Resource constraints
- Scheduling conflicts
